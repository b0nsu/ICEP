스터디룸 예약 관리 CLI는 회원가입과 로그인 후 스터디룸의 예약 가능 여부를 조회하고, 예약을 신청·취소하며, 예약 시작 시점에 체크인할 수 있는 문자 기반 프로그램이다. 관리자는 전체 예약 현황과 체크인 여부를 조회할 수 있고, 공용 가상 현재 일시를 변경하거나 예약을 다른 방으로 이동시키고, 스터디룸의 최대 수용 인원과 운영 상태를 조정할 수 있다. 본 프로그램은 실제 운영체제 시각이 아니라 데이터 파일에 저장된 공유 가상 현재 일시를 기준으로 동작하며, 예약 상태의 자동 갱신·충돌 확인·영향 예약 처리 규칙을 명시적으로 다루는 것을 특징으로 한다.

\section{용어}
이 문서에서 사용할 주요 용어의 의미를 먼저 약속해 둔다. 아래 용어들 가운데 일부는 세간에서 널리 쓰이는 표현을 이 문서의 주제에 맞게 조금 더 좁게 사용하며, 일부는 이 문서에서만 쓰이는 의미를 새로 부여한다. 용어를 먼저 정해 두는 이유는 이후의 기능 설명, 데이터 규칙, 예외 상황 설명에서 동일한 단어가 항상 동일한 뜻으로 해석되도록 하기 위함이다.

\subsection{사용자 역할}
\textbf{회원(member)} 특정 이 문서에서 회원은 회원가입을 마치고 로그인하여 스터디룸을 예약·취소·조회·체크인할 수 있는 일반 사용자를 뜻한다. 회원은 자신의 예약에 대해서만 조작할 수 있으며, 다른 사용자의 예약을 변경하거나 시스템 시간을 바꾸는 권한은 없다.

\textbf{관리자(admin)} 특정 이 문서에서 관리자는 회원이 할 수 없는 전체 조회, 현재 시각 변경, 예약 조정, 룸 컨디션 관리 기능을 수행할 수 있는 사용자이다. 관리자는 관리자 계정으로 로그인했을 때에만 이러한 기능에 접근할 수 있다.

\subsection{시간과 상태 관련 용어}
\textbf{가상 현재 일시} 명확 프로그램 전체가 공유하는 단일 기준 시각. 실제 컴퓨터의 시계와는 무관하며, 데이터 파일 \texttt{system\_time.txt}에 저장된 값만을 현재 시각으로 사용한다. 프로그램이 실행 중이라고 해서 자동으로 흐르지 않으며, 관리자 기능에 의해서만 변경할 수 있고 과거로 되돌릴 수는 없다.

\textbf{체크인 가능 구간} 신규 예약 시작 시각 10분 전부터 시작 시각 15분 후까지의 닫힌 구간. 회원은 자신의 예약이 \texttt{RESERVED} 상태이고 해당 룸이 운영 중인 경우에만 이 구간 안에서 체크인할 수 있다.

\textbf{예약 상태} 명확 예약이 현재 어떤 단계에 있는지를 나타내는 값. 본 프로그램은 \texttt{RESERVED}, \texttt{CHECKED\_IN}, \texttt{COMPLETED}, \texttt{CANCELLED}, \texttt{NO\_SHOW}의 다섯 상태만을 사용한다. 상태 간 이동 규칙은 7.1절에서 다시 정리한다.

\textbf{노쇼(NO\_SHOW)} 명확 예약이 \texttt{RESERVED} 상태인 채로 체크인 마감 시각을 지나 자동 전환된 종료 상태. 노쇼는 어디까지나 예약 이력과 상태 기록의 일부이며, 벌점·정지·예약 제한과 같은 패널티를 발생시키지 않는다.

\subsection{자원과 충돌 관련 용어}
\textbf{스터디룸} 특정 예약의 대상이 되는 물리적 방. 각 룸은 룸 ID, 이름, 최대 수용 인원, 장비 목록, 운영 상태를 가진다. 운영 상태가 \texttt{CLOSED}인 룸은 신규 예약과 체크인의 대상이 될 수 없다.

\textbf{예약 충돌} 신규 두 예약의 시간 구간이 같은 자원 위에서 겹치는 경우. 이 문서에서는 시간 구간을 \([start, end)\)의 반열린 구간으로 보며, 두 구간이 \(start_1 < end_2\)이고 동시에 \(start_2 < end_1\)이면 서로 겹친다고 본다. 충돌 검사는 같은 룸에 대한 예약끼리뿐 아니라 같은 회원의 예약끼리도 적용된다.

\textbf{영향 예약} 신규 룸 휴업 또는 최대 수용 인원 감소와 같은 관리자 조정으로 인해 더 이상 그대로 유지할 수 없게 되는 미래의 \texttt{RESERVED} 예약. 영향 예약이 존재할 경우 관리자는 각 예약을 다른 룸으로 이동시키거나 취소하거나, 이번 룸 변경 전체를 취소해야 한다.

\subsection{데이터 파일 관련 용어}
\textbf{데이터 파일} 특정 프로그램이 읽고 쓰는 UTF-8 일반 텍스트 파일. 이 문서에서 데이터 파일이라고 부르는 대상은 다음 네 파일뿐이다. \texttt{users.txt}, \texttt{rooms.txt}, \texttt{reservations.txt}, \texttt{system\_time.txt}.

\textbf{필드 구분자} 명확 각 데이터 파일의 한 줄 안에서 필드를 나누는 문자 \softsep{}를 뜻한다. 별도 이스케이프 규칙은 없으므로, 저장되는 실제 데이터 내부에 필드 구분자 문자 자체가 들어가면 안 된다.

\textbf{문법 규칙} 특정 각 줄이 어떤 형식을 따라야 하는지를 정하는 규칙이다. 예를 들어 레코드 접두어, 필드 개수, 날짜·시각 표기 형식, 정규식 제약 등이 여기에 포함된다.

\textbf{의미 규칙} 특정 형식적으로는 맞더라도 데이터 전체의 의미상 만족해야 하는 추가 조건이다. 예를 들어 사용자 ID 중복 금지, 정원 초과 불가, 동일 회원 시간 중복 금지 등은 의미 규칙에 속한다.

\section{기본사항}

\subsection{작동 환경}
본 프로그램은 Java 17 이상이 설치된 일반적인 데스크톱 셸 환경을 가정한다. 운영체제는 Windows, macOS, Linux 중 어느 것이든 무방하되, UTF-8 인코딩으로 텍스트 파일을 읽고 쓸 수 있어야 하고, 표준 입출력을 이용한 문자 인터페이스를 지원해야 한다. 화면은 그래픽 사용자 인터페이스가 아니라 터미널 또는 콘솔 창 하나만을 사용한다.

또한 본 프로그램은 DBMS, 네트워크 서버, 브라우저, 모바일 앱 등 별도의 외부 런타임을 요구하지 않는다. 사용자는 프로그램 실행 디렉토리 아래의 \texttt{data/} 폴더를 통해 저장 데이터를 직접 열람하거나, 필요할 경우 텍스트 편집기로 내용을 확인할 수 있다.

\subsection{프로그램 구성}
프로그램은 크게 세 층위로 나누어 이해할 수 있다. 첫째는 사용자와 상호작용하는 프롬프트 계층이고, 둘째는 예약·충돌·상태 전이와 같은 규칙을 판단하는 비즈니스 계층이며, 셋째는 텍스트 파일을 읽고 쓰는 저장 계층이다. 사용자는 메뉴와 입력 프롬프트를 통해 기능을 호출하며, 각 기능은 데이터를 읽어 현재 상태를 확인한 뒤 규칙 검증을 통과하면 파일에 반영한다.

로그인 전에는 회원가입, 로그인, 종료만 가능하다. 로그인 후에는 역할에 따라 \texttt{member} 메뉴와 \texttt{admin} 메뉴로 갈라진다. \texttt{member} 메뉴는 자기 예약 중심 기능만 제공하고, \texttt{admin} 메뉴는 시스템 수준 조회와 조정 기능을 제공한다.

\subsection{프로그램 설치 및 실행}
프로젝트 루트에서 빌드 스크립트를 실행한 뒤 프로그램을 시작하는 것을 기본 사용 방식으로 한다. 실행 방법의 한 예시는 아래와 같다.

\begin{menubox}
\small
\texttt{./build.sh}\par
\texttt{./run.sh}
\end{menubox}

빌드 결과가 JAR 파일로 생성되는 경우에는 직접 실행도 가능하다.

\begin{menubox}
\small
\texttt{java -jar out/study-room-cli.jar}
\end{menubox}

프로그램은 시작 시 \texttt{data/} 폴더와 필수 파일 네 개의 존재 여부를 먼저 확인한다. 파일이 누락되어 있으면 기본 내용을 가진 초기 파일을 자동 생성하고, 이미 존재한다면 문법과 의미를 순서대로 검사한 뒤 진행한다. 데이터 파일의 오류가 발견되면 파일명·줄번호·원인을 출력하고 시작을 중단한다.

그림 1에는 핵심 경로만 실었다. 실제로는 대부분의 기능 진입 직전에 현재 데이터 재로딩과 상태 자동 갱신이 한 번 더 수행된다. 이 때문에 회원이 예약을 신청하거나 관리자가 룸 상태를 변경하려 할 때, 이전 기능 수행 결과 또는 직전의 현재 시각 변경 결과가 항상 반영된 상태에서 검사가 이루어진다.

\section{데이터 요소}
이 프로그램이 직접 다루는 중심 데이터 요소는 사용자, 스터디룸, 예약, 가상 현재 일시의 네 종류이다. 각 데이터 요소는 별도의 파일 레코드로 저장되며, 일부 요소는 다른 요소를 참조한다. 예를 들어 예약은 반드시 실제 존재하는 사용자와 룸을 가리켜야 하며, 룸의 현재 정원은 예약 인원 검증에 사용된다.

\subsection{사용자}
사용자는 \texttt{userId}, \texttt{password}, \texttt{displayName}, \texttt{role}, \texttt{status}의 다섯 필드를 가진다. \texttt{userId}는 시스템 내부 식별자이며 중복될 수 없다. \texttt{displayName}은 사람에게 보여 주는 이름이므로 동일할 수 있지만, 동일한 \texttt{userId}는 다시 사용할 수 없다.

역할은 \texttt{member}와 \texttt{admin} 두 가지뿐이며, 회원가입으로 새로 만들어지는 계정의 역할은 항상 \texttt{member}이다. 상태는 현재 범위에서는 \texttt{ACTIVE} 하나만 사용한다. 즉, 로그인 가능 여부는 곧 활성 계정 여부와 같다.

\subsection{스터디룸}
스터디룸은 \texttt{roomId}, \texttt{roomName}, \texttt{maxCapacity}, \texttt{equipmentList}, \texttt{roomStatus}를 가진다. \texttt{roomId}는 예를 들어 \texttt{R101}과 같은 형식의 고유 식별자이고, \texttt{roomName}은 화면에 표시되는 방 이름이다. \texttt{maxCapacity}는 해당 룸에 허용되는 최대 이용 인원이며 1 이상의 정수만 가능하다.

\subsubsection{장비 목록}
\texttt{equipmentList}는 \texttt{PROJECTOR+WHITEBOARD}와 같이 \texttt{+}기호로 이어 붙인 토큰 목록 또는 장비 없음의 의미인 \texttt{-} 하나로 저장한다. 장비 목록은 예약 가능 여부 자체를 결정하지는 않지만, 예약 가능 룸 조회 결과를 사용자에게 설명하는 데 사용된다.

\subsection{예약}
예약은 \texttt{reservationId}, \texttt{userId}, \texttt{roomId}, \texttt{date}, \texttt{startTime}, \texttt{endTime}, \texttt{partySize}, \texttt{status}, \texttt{createdAt}, \texttt{checkedInAt}의 열 항목으로 저장된다. 예약 번호는 \texttt{rv0001} 형식으로 자동 부여되고, 같은 번호를 두 예약이 공유할 수 없다.

예약의 핵심 의미 요소는 이용 날짜, 시작 시각, 종료 시각, 이용 인원, 그리고 상태이다. 이 가운데 시간 구간과 인원은 예약 가능 여부 판단, 충돌 검출, 룸 이동 가능 여부 판단, 정원 변경 시 영향 예약 계산의 중심 기준이 된다. 예약 길이는 1시간 이상 4시간 이하이며, 시작·종료 시각은 모두 30분 단위여야 한다.

\begin{center}
\begin{tabularx}{0.94\linewidth}{>{\ttfamily\small}l X}
\toprule
상태 & 의미 \\
\midrule
RESERVED & 예약 생성은 완료되었으나 아직 체크인하지 않은 상태 \\
CHECKED\_IN & 체크인 가능 구간 안에서 체크인을 마친 상태 \\
COMPLETED & 체크인된 예약의 종료 시각이 지나 이용이 끝난 상태 \\
CANCELLED & 예약 시작 전 회원이 취소했거나 영향 예약 처리에서 취소된 상태 \\
NO\_SHOW & 체크인하지 않은 채 체크인 마감 시각을 지나 자동 종료된 상태 \\
\bottomrule
\end{tabularx}

표 1: 예약 상태의 종류
\end{center}

\subsection{가상 현재 일시}
가상 현재 일시는 별도의 독립 레코드 \texttt{NOW\softsep{}yyyy-MM-dd HH:mm} 형태로 저장된다. 이 값은 예약 생성 시 \texttt{createdAt} 기록, 체크인 허용 여부 판단, 예약 취소 가능 여부 판단, 자동 상태 전환의 기준으로 사용된다. 본 문서의 다른 모든 시간 규칙은 특별히 달리 적지 않는 한 이 가상 현재 일시와 비교한 결과를 뜻한다.

\section{데이터 파일}

\subsection{문법 규칙 (형식)}
네 데이터 파일은 모두 UTF-8 일반 텍스트 파일이며, 각 줄은 하나의 레코드이다. 빈 줄은 허용되며 무시할 수 있고, \texttt{\#}로 시작하는 줄은 주석으로 보아 무시한다. 각 레코드의 필드는 \softsep{} 문자로 구분한다. 별도의 이스케이프 규칙을 두지 않으므로, 실제 값 내부에 개행 문자, 캐리지 리턴, 필드 구분자 \softsep{}를 포함해서는 안 된다.

\subsubsection{설명과 예시}
각 파일의 기본 레코드 형식은 다음과 같다.

\begin{menubox}
\small
\texttt{users.txt}\par
\texttt{USER\softsep{}userId\softsep{}password\softsep{}displayName\softsep{}role\softsep{}status}\par\medskip
\texttt{rooms.txt}\par
\texttt{ROOM\softsep{}roomId\softsep{}roomName\softsep{}maxCapacity\softsep{}equipmentList\softsep{}roomStatus}\par\medskip
\texttt{reservations.txt}\par
\texttt{RESV\softsep{}reservationId\softsep{}userId\softsep{}roomId\softsep{}date\softsep{}startTime\softsep{}endTime\softsep{}partySize\softsep{}status\softsep{}createdAt\softsep{}checkedInAt}\par\medskip
\texttt{system\_time.txt}\par
\texttt{NOW\softsep{}2026-03-20 09:00}
\end{menubox}

문법적으로 올바른 예시는 아래와 같다.

\begin{menubox}
\small
\texttt{USER\softsep{}admin\softsep{}admin1234\softsep{}관리자\softsep{}admin\softsep{}ACTIVE}\par
\texttt{ROOM\softsep{}R101\softsep{}A룸\softsep{}4\softsep{}PROJECTOR+WHITEBOARD\softsep{}OPEN}\par
\texttt{RESV\softsep{}rv0007\softsep{}user011\softsep{}R101\softsep{}2026-03-20\softsep{}13:00\softsep{}15:00\softsep{}2\softsep{}RESERVED\softsep{}2026-03-20 09:10\softsep{}-}\par
\texttt{NOW\softsep{}2026-03-20 15:20}
\end{menubox}

\subsection{의미 규칙 (추가 조건)}
문법 규칙을 만족하는 것만으로는 충분하지 않다. 데이터 전체가 아래 의미 조건도 만족해야만 정상 데이터로 간주된다.

\begin{itemize}[leftmargin=1.8em,itemsep=0.25em]
\item 사용자 ID, 룸 ID, 예약 번호는 각각 파일 전체에서 유일해야 한다.
\item 전체 사용자 가운데 \texttt{role=admin}인 계정이 최소 1개 이상 존재해야 한다.
\item 예약 레코드의 \texttt{userId}와 \texttt{roomId}는 반드시 실제 존재하는 사용자와 룸을 가리켜야 한다.
\item 예약의 시작 시각은 종료 시각보다 빨라야 하며, 예약 길이는 60분 이상 240분 이하이어야 한다.
\item 예약 인원은 1 이상이어야 한다.
\item \texttt{RESERVED} 또는 \texttt{CHECKED\_IN} 상태의 예약은 해당 룸의 현재 최대 수용 인원을 초과할 수 없다.
\item \texttt{CHECKED\_IN}, \texttt{COMPLETED} 상태는 \texttt{checkedInAt} 값이 실제 일시여야 하고, 나머지 상태는 \texttt{-}이어야 한다.
\item 같은 룸의 \texttt{RESERVED} 또는 \texttt{CHECKED\_IN} 예약끼리는 시간 충돌이 있으면 안 된다.
\item 같은 회원의 \texttt{RESERVED} 또는 \texttt{CHECKED\_IN} 예약끼리도 시간 충돌이 있으면 안 된다.
\item \texttt{system\_time.txt}에는 유효한 \texttt{NOW} 레코드가 정확히 하나만 있어야 한다.
\end{itemize}

\subsection{부가 확인 항목}
의미 규칙 외에도 프로그램의 정상 동작을 위해 추가 확인이 필요한 항목이 있다. 이 항목들은 저장 형식의 문법과는 다르지만, 기능 수행의 허용·거부 판단에서 반복적으로 사용된다.

\subsubsection{예약 충돌 확인}
예약 가능 스터디룸 조회, 예약 신청, 예약 조정(방 이동), 영향 예약 처리에서 공통으로 사용하는 검사 항목이다. 같은 날짜에 같은 룸을 사용하는 두 예약이 시간 구간 \([start, end)\) 기준으로 겹치면 둘은 공존할 수 없다. 회원 자신의 예약끼리도 같은 방식으로 충돌을 검사한다.

\subsubsection{영향 예약}
관리자가 룸을 \texttt{CLOSED}로 바꾸거나 정원을 줄이려 할 때, 이미 존재하는 미래 \texttt{RESERVED} 예약이 계속 유효한지 재판정해야 한다. 새 정원을 초과하는 예약, 또는 더 이상 운영 중이 아닌 룸에 놓이게 되는 예약은 영향 예약으로 간주한다. 영향 예약이 하나라도 있으면 단순 반영으로 끝나지 않고 별도의 처리 절차로 넘어간다.

\subsection{무결성 확인 및 처리}
프로그램 시작 시와 주요 기능 수행 직전에는 데이터 전체를 다시 로드하고, 위 문법 및 의미 규칙을 차례로 검사한다. 오류가 발견되면 조용히 무시하지 않고 파일명과 줄번호, 원인을 함께 출력한 후 작업을 중단한다. 파일이 아예 존재하지 않는 경우에는 오류로 끝내지 않고 기본 파일을 생성한다.

또한 현재 시각 변경 이후에는 예약 상태를 전부 다시 계산하여, \texttt{RESERVED}가 \texttt{NO\_SHOW}로, \texttt{CHECKED\_IN}이 \texttt{COMPLETED}로 자동 전환되어야 하는지 점검한다. 이때 실제로 바뀐 레코드가 있다면 다시 저장하고 요약 정보를 보여 준다.

\section{주 프롬프트}
이 프로그램의 주 프롬프트는 로그인 여부와 역할에 따라 제공되는 메뉴가 달라지는 계층형 문자 인터페이스이다. 가장 바깥에는 비로그인 메뉴가 있고, 로그인 후에는 \texttt{member} 또는 \texttt{admin} 메뉴가 열린다. 일부 기능은 추가 입력을 받는 부 프롬프트로 이어진다.

\subsection{비로그인 메뉴군}
비로그인 상태에서 사용자는 회원가입, 로그인, 종료만 선택할 수 있다.

\begin{menubox}
\small
[비로그인 메뉴]\par
1. 회원가입\par
2. 로그인\par
0. 종료
\end{menubox}

\subsubsection{부 프롬프트: 회원가입 입력}
회원가입은 \texttt{userId}, 비밀번호, 표시 이름을 차례대로 입력받아 새로운 \texttt{member} 계정을 만드는 과정이다. \texttt{userId}는 영문자로 시작하고 영문자·숫자·밑줄만을 사용하여 4자 이상 20자 이하로 입력해야 한다. 비밀번호는 4자 이상 20자 이하, 표시 이름은 1자 이상 20자 이하로 제한한다.

중복된 사용자 ID가 들어오거나 허용되지 않는 문자가 포함되어 있으면 오류 메시지를 출력하고 같은 입력 단계를 다시 받는다. 성공 시에는 \texttt{users.txt}에 새 \texttt{USER} 레코드를 추가한다.

\subsubsection{부 프롬프트: 로그인 입력}
로그인은 사용자 ID와 비밀번호를 입력받아 계정을 인증하는 과정이다. 입력된 ID가 존재하지 않거나 비밀번호가 일치하지 않으면 로그인은 실패하며, 성공한 경우 계정의 \texttt{role} 값에 따라 \texttt{member} 메뉴 또는 \texttt{admin} 메뉴로 진입한다.

\subsection{member 메뉴군}
\texttt{member} 메뉴는 일반 회원이 자신의 예약을 중심으로 사용하는 메뉴이다.

\begin{menubox}
\small
[member 메뉴]\par
1. 현재 가상 시각 조회\par
2. 예약 가능 스터디룸 조회\par
3. 예약 신청\par
4. 예약 취소\par
5. 나의 예약 조회\par
6. 체크인\par
0. 로그아웃
\end{menubox}

\subsubsection{현재 가상 시각 조회}
현재 가상 시각 조회는 프로그램 전체가 기준으로 삼는 공용 시각을 보여 주는 기능이다. 출력값은 \texttt{system\_time.txt}에 저장된 \texttt{yyyy-MM-dd HH:mm} 형식의 값이며, 회원은 이를 통해 예약 신청 가능 여부, 예약 취소 가능 여부, 체크인 가능 구간 진입 여부를 판단할 수 있다. 이 기능은 조회만 수행하므로 어떠한 데이터도 변경하지 않는다.

\subsubsection{예약 가능 스터디룸 조회}
이 기능은 날짜, 시작 시각, 종료 시각, 이용 인원을 입력받아 조건을 만족하는 룸 목록을 보여 준다. 출력 대상은 \texttt{OPEN} 상태이며, 입력 인원을 수용할 수 있고, 해당 시간대에 다른 예약과 충돌하지 않는 룸들이다. 이 기능은 단지 조회 기능이므로, 어떠한 저장 데이터도 변경하지 않는다.

\subsubsection{예약 신청}
회원은 날짜, 시작 시각, 종료 시각, 이용 인원, 룸 ID를 입력하여 예약을 신청할 수 있다. 예약 신청 시에는 다음 항목을 모두 검사한다.

\begin{itemize}[leftmargin=1.8em,itemsep=0.2em]
\item 예약 길이가 1시간 이상 4시간 이하인지
\item 시작 시각과 종료 시각이 30분 단위인지
\item 시작 시각이 종료 시각보다 빠른지
\item 대상 룸이 존재하고 \texttt{OPEN} 상태인지
\item 이용 인원이 룸 정원 이내인지
\item 예약 시작 일시가 현재 가상 시각보다 미래인지
\item 같은 룸의 같은 시간대에 기존 예약이 없는지
\item 같은 회원의 다른 예약과 시간이 겹치지 않는지
\end{itemize}

성공 시 프로그램은 다음 예약 번호를 자동 생성하고, 현재 가상 시각을 \texttt{createdAt}으로 기록하며, 상태를 \texttt{RESERVED}로 저장한다.

\subsubsection{예약 취소}
회원은 자신의 \texttt{RESERVED} 예약에 대해서만 취소를 시도할 수 있다. 또한 취소 시점이 예약 시작 시각 이전이어야 한다. 이미 시작되었거나, 체크인되었거나, 종료 상태인 예약은 회원이 취소할 수 없다. 성공하면 상태를 \texttt{CANCELLED}로 바꾼다.

\subsubsection{나의 예약 조회}
로그인한 회원의 예약만을 날짜, 시작 시각, 룸 ID, 예약 번호 순으로 정렬하여 보여 준다. 각 행에는 예약 번호, 룸 ID, 룸 이름, 날짜, 시작 시각, 종료 시각, 이용 인원, 현재 상태를 포함한다. 노쇼나 취소도 이 조회 화면에서 그대로 드러난다.

\subsubsection{체크인}
체크인은 자신의 \texttt{RESERVED} 예약에 대해서만 가능하다. 현재 시각이 체크인 가능 구간인 예약 시작 10분 전부터 시작 후 15분까지에 속해야 하며, 해당 룸은 여전히 \texttt{OPEN} 상태여야 한다. 성공하면 상태는 \texttt{CHECKED\_IN}으로, \texttt{checkedInAt}은 현재 가상 시각으로 갱신된다.

\subsection{admin 메뉴군}
\texttt{admin} 메뉴는 시스템 전체 상태를 조회하고 조정하는 메뉴이다.

\begin{menubox}
\small
[admin 메뉴]\par
1. 현재 가상 시각 조회\par
2. 현재 가상 시각 변경\par
3. 전체 예약 정보 조회\par
4. 전체 체크인 여부 조회\par
5. 예약 조정(방 이동)\par
6. 룸 컨디션 관리\par
0. 로그아웃
\end{menubox}

\subsubsection{현재 가상 시각 조회}
현재 가상 시각 조회는 \texttt{system\_time.txt}의 값을 그대로 보여 주는 기능이다. \texttt{member} 메뉴의 동일 기능과 출력 대상은 같지만, 관리자는 이 값을 확인한 뒤 곧바로 다음 기능인 현재 가상 시각 변경으로 이어서 진입할 수 있다는 점이 다르다.

\subsubsection{현재 가상 시각 변경}
현재 가상 시각 변경은 새 \texttt{yyyy-MM-dd HH:mm} 값을 입력받아 기존 값과 비교한 뒤, 과거로 되돌리는 입력은 거부하고 같거나 미래인 경우에만 반영한다. 변경 성공 후에는 전체 예약 상태를 즉시 재판정하여, \texttt{RESERVED -> NO\_SHOW}, \texttt{CHECKED\_IN -> COMPLETED} 등 실제 전환이 발생한 건수를 요약 출력한다. 변화가 없으면 상태 변화 없음 메시지를 출력한다.

\subsubsection{전체 예약 정보 조회}
전체 예약 정보 조회는 모든 예약을 날짜, 시작 시각, 룸 ID, 예약 번호 순으로 정렬하여 출력하는 기능이다. 예약 번호, 사용자 ID, 룸 ID, 날짜, 시작·종료 시각, 인원 수, 현재 상태, 체크인 시각을 함께 보여 줌으로써 시스템 전체 예약 상황을 한눈에 파악하도록 한다.

\subsubsection{전체 체크인 여부 조회}
전체 체크인 여부 조회는 전체 예약을 같은 정렬 기준으로 훑되, 체크인 관련 상태 확인에 필요한 핵심 열 위주로 간략하게 보여 주는 기능이다. 특히 \texttt{CHECKED\_IN}, \texttt{COMPLETED}, \texttt{NO\_SHOW}, \texttt{RESERVED}, \texttt{CANCELLED}의 차이를 빠르게 확인하는 데 목적이 있다.

\subsubsection{예약 조정(방 이동)}
예약 조정은 미래의 \texttt{RESERVED} 예약 하나를 선택하여, 날짜·시간·인원·예약자 정보는 유지한 채 룸만 다른 룸으로 바꾸는 기능이다. 대상 룸은 반드시 존재해야 하고 \texttt{OPEN} 상태여야 하며, 현재 룸과 다른 룸이어야 하고, 기존 예약 인원을 수용할 수 있어야 하며, 같은 시간대에 다른 예약과 충돌하지 않아야 한다. 이 조건을 하나라도 만족하지 않으면 이동은 실패한다.

\subsubsection{룸 컨디션 관리}
룸 컨디션 관리에는 전체 룸 조회, 최대 수용 인원 변경, 임시 휴업, 운영 재개가 포함된다.

\begin{menubox}
\small
[룸 컨디션 관리]\par
1. 전체 룸 조회\par
2. 룸 최대 수용 인원 변경\par
3. 룸 임시 휴업\par
4. 룸 운영 재개\par
0. 상위 메뉴로
\end{menubox}

전체 룸 조회는 각 룸의 ID, 이름, 정원, 장비 목록, 운영 상태를 그대로 보여 준다. 최대 수용 인원 변경은 새 정원을 초과하는 미래 \texttt{RESERVED} 예약이 있으면 먼저 영향 예약 처리 절차를 수행하며, 처리 후에도 활성 예약이 새 정원을 넘으면 최종 변경을 거부한다. 임시 휴업은 현재 \texttt{CHECKED\_IN} 예약이 있는 룸에 대해서는 곧바로 적용할 수 없다. 운영 재개는 룸 상태를 다시 \texttt{OPEN}으로 돌리는 기능이다.

관리자가 정원을 줄이거나 룸을 휴업하려 할 때 미래 \texttt{RESERVED} 예약에 영향이 생기면, 단순히 룸 상태만 바꾸지 않고 영향 예약 처리 절차로 들어간다. 이 절차에서는 각 영향 예약마다 다른 룸으로 이동, 해당 예약 취소, 이번 변경 전체 취소 중 하나를 선택한다. 모든 영향 예약을 정상 처리한 뒤에만 최종 룸 변경이 확정된다.

\section{기타 처리}

\subsection{상태 자동 갱신}
상태 자동 갱신은 프로그램이 시작할 때와 현재 시각 변경 직후, 그리고 주요 기능 수행 직전에 반복적으로 수행되는 핵심 처리이다. 규칙은 단순하다. \texttt{RESERVED} 상태에서 현재 시각이 예약 시작 시각보다 15분을 초과하면 \texttt{NO\_SHOW}로 바뀐다. \texttt{CHECKED\_IN} 상태에서 현재 시각이 종료 시각 이상이 되면 \texttt{COMPLETED}로 바뀐다. \texttt{CANCELLED}, \texttt{COMPLETED}, \texttt{NO\_SHOW}는 종료 상태로 간주하며 다시 되돌리지 않는다.

\subsection{파일 누락 시 초기 데이터 생성}
필수 파일이 하나라도 없으면 프로그램은 \texttt{data/} 폴더를 만들고 기본 파일을 자동 생성한다. 이때 \texttt{users.txt}에는 \texttt{USER\softsep{}admin\softsep{}admin1234\softsep{}관리자\softsep{}admin\softsep{}ACTIVE} 한 줄이 들어가고, \texttt{rooms.txt}에는 \texttt{R101/A룸/4}, \texttt{R102/B룸/6}, \texttt{R103/C룸/8}에 해당하는 세 개의 \texttt{OPEN} 룸 레코드가 생성된다. \texttt{system\_time.txt}는 \texttt{NOW\softsep{}2026-03-20 09:00}으로 초기화되며, \texttt{reservations.txt}는 빈 파일로 시작한다.

\subsection{오류 메시지와 재입력}
이 프로그램은 잘못된 입력이나 잘못된 데이터를 조용히 무시하지 않는다. 메뉴 번호가 잘못되면 해당 메뉴 단계에서 다시 입력받고, 회원가입·로그인·예약 입력 과정에서 형식이 틀리면 구체적인 이유를 함께 출력한 뒤 같은 항목을 다시 입력받는다. 데이터 파일 오류의 경우에는 파일명, 줄번호, 문제 원인을 함께 제시하여 사용자가 스스로 수정할 수 있게 한다.

\subsection{파일 직접 수정 시 처리}
사용자가 프로그램을 종료한 상태에서 텍스트 편집기로 데이터 파일을 직접 수정하는 것은 허용된다. 다만 수정된 내용은 다음 실행 시점에 다시 문법·의미 검사를 통과해야 하며, 검사에 실패하면 프로그램은 해당 오류를 출력하고 시작을 중단한다. 프로그램이 실행 중인 동안 외부 편집기로 파일을 동시에 수정하는 경우에는 정상 동작을 보장하지 않는다.
