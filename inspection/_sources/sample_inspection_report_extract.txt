정적 분석static analysis: 프로그램을 실제로 실행시키지 않은 채로 (주로 소스 코드를) 분석하는 방법

예: 정형 검증formal verification
• 분석 대상(주로 소스 코드)과 속성을 수식/논리식으로 표현하여, 대상이 속성을 만족시키는지
수리적/논리적으로 증명하는 분석 방법
• 분석 결과의 신뢰성 높지만, 시간과 비용이 많이 들며 분석 가능 범위에 제약이 있음
동적 분석dynamic analysis: 프로그램 일부나 전체를 실제로 실행시켜보면서 분석하는 방법
실행에 추가 장비가 필요 없고 분석 중 오류 발생 시 피해가 적은 경우 유용
예: 검사testing
• 프로그램/함수 등에 여러가지 입력/인자를 넣고 실행/호출해보아 그 실제 결과가 예상 결과와 같은지
확인하여, 프로그램의 속성을 확률적으로 추정하는 분석 방법
• 분석 신뢰성이 (상대적으로) 다소 낮지만, 빠르고 저비용이며 분석 가능 범위의 제약도 훨씬 적음

단위 검사unit test
대상: 루프 블럭, 조건 분기, 서브루틴, 프로시져, 함수 등
입력 방법: 대상에 인자, 변수값, 상태 등을 강제로 (검사 후에는 없앨 임시 대입/호출문으로) 만들어 넣기
결과 확인: 대상의 리턴값, 변수값, 상태 등을 강제로 (검사 후에는 없앨 임시 출력문으로) 찍어보기
비유: 선풍기를 만들 때
• 모터가 잘 돌아가는지 임시 배터리를 모터에 직접 붙여서 확인
• 내부 회로에 신호가 잘 전달되는지 전극을 대보고 확인
통합 검사integration test
대상: 단위 검사를 마친 부분들을 서로 연동시켜 합친 큰 덩어리
각 단위를 블랙박스로 간주하여 그 내부를 건드리지 않고 검사
대상이 (중간 단계 덩어리가 아니라) 최종 완제품일 경우 “전체 통합 검사”
• 프로그램의 외부 (즉, 사용자) 입력/조작만으로 테스트 케이스(후술) 구성
• 프로그램의 외부 출력/동작만으로 실제 결과 확인
개발자가 직접 하기도 하고, (특히 전체 통합 검사는) 모집된 흑우베타 테스터들이 해주기도 함
비유: 선풍기를 전혀 분해하지 않고 그냥 콘센트에 연결하고 조작 버튼을 눌러서 동작 확인

테스트 케이스(후술) 기반 검사
기본 개념: 충분한 범위와 분량(후술) 의 테스트 케이스들을 준비해서, 각 테스트 케이스마다 그에 대응하는
‘실제 결과’를 확인하여 이를 각각의 ‘예상 결과’와 대조하는 검사 방식
계획 주도 개발에서: 일단 계획부터 완벽하게 마무리해둔 후, 그에 맞춰 개발하고, 계획 대로인지 검사
• 계획 중 일부가 틀어지면 계획 수정부터 다시 진행
• 계획 (기획/설계) → 구현 → 테스트 케이스 작성 → 검사
테스트 주도 개발에서: 필요(요구)에 따라 테스트 케이스부터 먼저 만들고, 이를 만족하도록 개발
• ‘완벽한 계획’에 목 매지 않음: 계획 중 일부가 틀어져도 그 요소만 테스트 케이스 추가
• 계획 (기획/설계) 겸 테스트 케이스 작성 → 구현 → 검사
즉흥 검사ad hoc testing
테스트 케이스를 미리 작성하지 않고, 떠오르는 족족 임의로 검사
심지어 ‘예상 결과’조차 미리 준비해두지 않음: 실제 결과를 본 후 그 결과가 적절한지 판정
주로 ‘테스트 케이스 기반 검사’ 완료 후 추가로 진행 (초소형 프로젝트에서는 단독으로도 진행)
