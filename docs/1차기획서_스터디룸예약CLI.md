# 가상 현재 일시 기반 스터디룸 예약·이용 관리 시스템 1차 기획서 초안

본 문서는 Java CLI 프로젝트 **가상 현재 일시 기반 스터디룸 예약·이용 관리 시스템**의 1차 기획서 초안이다. 본 문서에서 정한 규칙은 1차 구현 범위에서 확정 규칙으로 사용한다.

## 1. 프로젝트명

가상 현재 일시 기반 스터디룸 예약·이용 관리 시스템 (Java CLI)

## 2. 프로젝트 개요

본 시스템은 여러 사용자가 로그인하여 같은 스터디룸 자원을 공유하는 예약·이용 관리 프로그램이다. 프로그램은 실제 컴퓨터 시간을 사용하지 않고, 모든 사용자가 함께 보는 **공용 가상 현재 일시**를 기준으로 동작한다. 사용자는 스터디룸을 검색하고 예약하며, 예약 당일에는 체크인과 1회 연장을 수행할 수 있다. 관리자는 전체 예약을 조회하고, 사용자 패널티를 초기화하며, 룸 운영 상태를 변경한다.

데이터 저장은 JSON, CSV, DBMS, 직렬화 기능을 사용하지 않고 팀이 정의한 일반 텍스트 파일만 사용한다. 사용자가 메모장 등으로 파일을 직접 수정할 수 있다고 가정하며, 이 경우의 문법 오류와 의미 오류를 감지하는 정책도 포함한다.

## 3. 목표와 비목표

### 3.1 목표

- Java 기반 CLI 환경에서 동작하는 다중 사용자 예약 시스템을 구현한다.
- `member`, `admin` 두 역할을 구분하고 권한을 분리한다.
- 실제 시간 대신 공용 가상 현재 일시를 사용한다.
- 스터디룸 예약, 취소, 체크인, 연장, 패널티 관리 규칙을 명확히 고정한다.
- 일반 텍스트 파일만으로 데이터를 저장하고 다시 읽어들인다.
- 파일 직접 수정으로 인한 오류를 문법 오류와 의미 오류로 나누어 감지한다.
- 1차 구현 범위를 무리 없이 완성할 수 있도록 핵심 기능만 우선 확정한다.

### 3.2 비목표

- 실제 운영 배포를 위한 보안 강화, 암호화 저장, 네트워크 연동은 1차 범위에 포함하지 않는다.
- GUI, 화살표키 기반 TUI, 모바일 앱, 웹 프런트엔드는 포함하지 않는다.
- 외부 API 호출, 크롤링, 실시간 알림, 이메일 발송은 포함하지 않는다.
- 멀티스레딩, IPC, 분산 처리, 동시 접속 제어는 포함하지 않는다.
- 대기열, 통계, 날짜별 점검 시간 등록, 세부 검색 고도화는 2차 확장 범위로 분리한다.
- 실제 시스템 시계와 동기화되는 기능은 포함하지 않는다.

## 4. 사용자 역할

### 4.1 member

- 회원가입 가능
- 로그인 가능
- 현재 시각 조회 및 변경 가능
- 스터디룸 목록 조회 가능
- 조건 기반 사용 가능 룸 검색 가능
- 예약 생성 가능
- 내 예약 조회 가능
- 예약 취소 가능
- 체크인 가능
- 사용 연장 가능
- 본인 패널티 조회 가능

제약은 다음과 같다.

- 미래 예약은 최대 2개까지만 가질 수 있다.
- 패널티가 2점 이상이면 새로운 예약을 생성할 수 없다.
- 같은 시간대에 본인 예약이 겹치면 예약할 수 없다.

### 4.2 admin

- 회원가입 대상이 아니다.
- `users.txt`에 사전 등록된 계정으로만 로그인한다.
- 현재 시각 조회 및 변경 가능
- 스터디룸 목록 조회 가능
- 전체 예약 조회 가능
- 사용자 패널티 조회 및 초기화 가능
- 룸 운영 상태 변경 가능

admin은 1차 범위에서 스터디룸 예약 생성, 체크인, 연장, 취소 기능을 사용하지 않는다. 따라서 admin 계정의 패널티는 항상 0으로 유지한다.

## 5. 주요 용어 정의

- **가상 현재 일시**: 실제 컴퓨터 시간이 아니라 `system_time.txt`에 저장된 공용 기준 시각이다.
- **공용 시각**: 로그인한 모든 사용자가 함께 사용하는 단일 현재 시각이다.
- **운영 시간**: 특정 룸이 매일 예약을 받을 수 있는 일일 시작 시각과 종료 시각이다.
- **운영 상태**: 룸의 현재 사용 가능 여부를 나타내는 값이며 `OPEN`, `CLOSED`, `MAINTENANCE` 중 하나다.
- **미래 예약**: 현재 가상 현재 일시보다 시작 시각이 늦고, 상태가 `RESERVED`인 예약이다.
- **예약 겹침**: 두 예약의 시간 구간이 한 분 이상 겹치는 경우다. 종료 시각과 다른 예약의 시작 시각이 정확히 같으면 겹치지 않는다.
- **체크인 가능 구간**: 예약 시작 10분 전부터 예약 시작 후 15분까지의 구간이며 양 끝값을 모두 포함한다.
- **자유 취소**: 예약 시작 30분 전까지 수행한 취소이며 패널티가 없다.
- **지연 취소**: 예약 시작 30분 전을 지난 후 `RESERVED` 상태에서 수행한 취소이며 패널티 1점을 부여한다.
- **노쇼(NO_SHOW)**: 체크인 가능 구간이 끝날 때까지 체크인하지 않은 예약 상태다.
- **연장**: `CHECKED_IN` 상태의 예약에 대해 종료 시각을 정확히 30분 늦추는 행위다.
- **자동 상태 갱신**: 가상 현재 일시를 기준으로 예약 상태를 `NO_SHOW`, `COMPLETED` 등으로 자동 변경하는 처리다.

## 6. 시스템 데이터와 저장 파일 개요

### 6.1 저장 파일 구성

본 프로젝트는 작업 디렉터리 기준으로 아래 네 개의 필수 텍스트 파일을 사용한다.

| 파일명 | 목적 | 기본 레코드 접두어 | 비고 |
|---|---|---|---|
| `users.txt` | 사용자 계정, 역할, 패널티 저장 | `USER` | 최소 1개의 admin 계정 필요 |
| `rooms.txt` | 룸 정보와 운영 상태 저장 | `ROOM` | 룸별 운영 시간 포함 |
| `reservations.txt` | 예약 이력과 현재 예약 상태 저장 | `RESV` | 체크인 시각, 연장 여부 포함 |
| `system_time.txt` | 공용 가상 현재 일시 저장 | `NOW` | 정확히 1개 레코드 |

### 6.2 공통 저장 정책

- 파일 인코딩은 UTF-8로 고정한다.
- 한 줄에 정확히 한 개의 레코드만 저장한다.
- 줄 구분은 LF를 기준으로 저장하고, 읽을 때는 LF와 CRLF를 모두 허용한다.
- 빈 줄은 허용하지 않는다.
- 주석 줄은 허용하지 않는다.
- 각 파일은 지정된 접두어 레코드만 허용한다.
- 필드 구분자는 `|` 하나만 사용한다.
- 모든 상태 변경 저장은 **임시 파일 작성 → 임시 파일 재검증 → 원본 교체** 순서로 처리한다.
- 저장 도중 오류가 발생하면 기존 원본 파일은 유지한다.

### 6.3 파일 직접 수정 대응 원칙

- 프로그램 시작 시 네 개 파일을 모두 읽고 문법 규칙과 의미 규칙을 검증한다.
- 시작 시 검증에 실패하면 프로그램은 어떤 파일도 수정하지 않고 종료한다.
- 상태를 바꾸는 기능을 실행하기 직전에는 디스크 파일을 다시 읽고 재검증한다.
- 재검증에 실패하면 현재 작업을 취소하고 어떤 파일도 수정하지 않은 채 프로그램을 종료한다.
- 부분 로딩, 부분 저장, 오류 레코드 무시는 허용하지 않는다.

## 7. 파일별 문법 규칙

### 7.1 공통 문법 요소

- **필드 구분자 규칙**: 모든 파일은 `|`를 필드 구분자로 사용한다.
- **이스케이프 규칙**: 텍스트 필드 안에서 `\|`는 문자 `|`, `\\`는 문자 `\`, `\n`은 줄바꿈 문자, `\r`은 캐리지리턴을 뜻한다. 이 네 가지 외의 역슬래시 조합은 문법 오류다.
- **공백 규칙**: 필드 내부 공백은 데이터로 취급한다. 다만 필드 앞뒤 공백은 저장하지 않는 것을 원칙으로 하며, 수동 편집으로 앞뒤 공백이 들어간 경우는 의미 오류로 처리한다.
- **날짜 형식 토큰**: 파일 내부 날짜는 `yyyy-MM-dd` 형식만 허용한다.
- **시각 형식 토큰**: 파일 내부 시각은 `HH:mm` 형식만 허용한다.
- **일시 형식 토큰**: 파일 내부 일시는 `yyyy-MM-ddTHH:mm` 형식만 허용한다.

### 7.2 `users.txt`

- **파일 목적**: 사용자 계정과 역할, 패널티를 저장한다.
- **한 줄 레코드 형식**: `USER|<userId>|<password>|<name>|<role>|<penalty>|<status>`
- **필드 구분자 규칙**: 7개 필드를 `|`로 정확히 구분한다.
- **이스케이프 규칙**: `userId`, `password`, `name` 필드에 공통 이스케이프 규칙을 적용한다.
- **문법 규칙**
  - 접두어는 반드시 `USER`다.
  - `userId`는 4자 이상 12자 이하이며 영문 소문자로 시작하고 이후에는 영문 소문자와 숫자만 허용한다.
  - `password`는 6자 이상 16자 이하이며 영문 대/소문자, 숫자, `!@#$%^&*._-`만 허용한다.
  - `name`은 2자 이상 20자 이하이며 한글, 영문, 단일 공백만 허용한다.
  - `role`은 `member` 또는 `admin`만 허용한다.
  - `penalty`는 0 이상 999 이하의 정수만 허용한다.
  - `status`는 1차 범위에서 `ACTIVE`만 허용한다.
- **예시 레코드**

```text
USER|admin01|Admin123!|관리자 계정|admin|0|ACTIVE
USER|u1001|pw1234|홍길동|member|0|ACTIVE
```

### 7.3 `rooms.txt`

- **파일 목적**: 룸 기본 정보, 비품, 운영 상태, 운영 시간을 저장한다.
- **한 줄 레코드 형식**: `ROOM|<roomId>|<capacity>|<equipmentList>|<status>|<openTime>|<closeTime>`
- **필드 구분자 규칙**: 7개 필드를 `|`로 정확히 구분한다.
- **이스케이프 규칙**: `equipmentList`를 제외한 텍스트 필드는 공통 이스케이프 규칙을 적용한다. `equipmentList`는 예약된 기호 체계만 허용하므로 이스케이프를 사용하지 않는다.
- **문법 규칙**
  - 접두어는 반드시 `ROOM`이다.
  - `roomId`는 `R` + 숫자 3자리 형식만 허용한다.
  - `capacity`는 1 이상 20 이하의 정수만 허용한다.
  - `equipmentList`는 `-` 또는 `EQUIP_CODE(+EQUIP_CODE)*` 형식만 허용한다.
  - `EQUIP_CODE`는 영문 대문자로 시작하며 영문 대문자, 숫자, `_`만 허용한다.
  - `status`는 `OPEN`, `CLOSED`, `MAINTENANCE`만 허용한다.
  - `openTime`, `closeTime`은 `HH:mm` 형식이며 분은 `00` 또는 `30`만 허용한다.
- **예시 레코드**

```text
ROOM|R101|4|PROJECTOR+WHITEBOARD|OPEN|09:00|23:00
ROOM|R202|8|-|OPEN|10:00|22:00
```

### 7.4 `reservations.txt`

- **파일 목적**: 예약 생성 결과와 이후의 상태 변화를 모두 저장한다.
- **한 줄 레코드 형식**: `RESV|<reservationId>|<roomId>|<userId>|<date>|<startTime>|<endTime>|<status>|<checkedInAt>|<extensionCount>`
- **필드 구분자 규칙**: 10개 필드를 `|`로 정확히 구분한다.
- **이스케이프 규칙**: `reservationId`, `roomId`, `userId`는 예약된 형식만 허용하므로 이스케이프를 사용하지 않는다. `checkedInAt`에는 공통 이스케이프를 사용하지 않는다.
- **문법 규칙**
  - 접두어는 반드시 `RESV`다.
  - `reservationId`는 `rv` + 숫자 4자리 이상 형식만 허용한다.
  - `roomId`는 `R` + 숫자 3자리 형식만 허용한다.
  - `userId`는 `users.txt`와 같은 형식을 사용한다.
  - `date`는 `yyyy-MM-dd` 형식만 허용한다.
  - `startTime`, `endTime`은 `HH:mm` 형식이며 분은 `00` 또는 `30`만 허용한다.
  - `status`는 `RESERVED`, `CANCELLED`, `CHECKED_IN`, `COMPLETED`, `NO_SHOW`만 허용한다.
  - `checkedInAt`은 `-` 또는 `yyyy-MM-ddTHH:mm` 형식만 허용한다.
  - `extensionCount`는 `0` 또는 `1`만 허용한다.
- **예시 레코드**

```text
RESV|rv0001|R101|u1001|2026-03-18|13:00|15:00|RESERVED|-|0
RESV|rv0002|R202|u1002|2026-03-10|18:00|20:30|CHECKED_IN|2026-03-10T17:55|1
```

### 7.5 `system_time.txt`

- **파일 목적**: 모든 사용자가 공유하는 현재 가상 일시를 저장한다.
- **한 줄 레코드 형식**: `NOW|<currentDateTime>`
- **필드 구분자 규칙**: 2개 필드를 `|`로 정확히 구분한다.
- **이스케이프 규칙**: `currentDateTime`은 고정 형식이므로 이스케이프를 사용하지 않는다.
- **문법 규칙**
  - 접두어는 반드시 `NOW`다.
  - `currentDateTime`은 `yyyy-MM-ddTHH:mm` 형식만 허용한다.
  - 파일에는 정확히 한 줄만 존재해야 한다.
- **예시 레코드**

```text
NOW|2026-03-10T13:00
```

## 8. 파일별 의미 규칙

### 8.1 `users.txt`

- `userId`는 파일 전체에서 유일해야 한다.
- 최소 1개 이상의 `admin` 계정이 존재해야 한다.
- `admin` 계정의 `penalty`는 항상 0이어야 한다.
- `admin` 계정은 예약 기능을 사용하지 않으므로 `reservations.txt`에서 참조할 수 없다.
- `member` 계정의 `penalty`는 누적 점수이며 자동 감소하지 않는다.
- `status`는 1차 범위에서 모두 `ACTIVE`여야 한다.
- 이름은 비어 있을 수 없고, 앞뒤 공백과 연속 공백 두 칸 이상을 허용하지 않는다.
- 회원가입으로 생성되는 계정은 항상 `member`다.
- **잘못된 파일 대응 방식**: 중복 `userId`, admin 부재, admin 패널티 비정상, 허용되지 않은 상태값이 발견되면 전체 파일을 무효로 판단하고 프로그램은 오류 메시지를 출력한 뒤 종료한다.

### 8.2 `rooms.txt`

- `roomId`는 파일 전체에서 유일해야 한다.
- `capacity`는 1 이상 20 이하이어야 한다.
- `equipmentList`의 장비 코드는 중복될 수 없다.
- `openTime`은 `closeTime`보다 빨라야 한다.
- `openTime`과 `closeTime`은 매일 동일하게 적용되는 일일 운영 시간이다.
- `status`가 `CLOSED` 또는 `MAINTENANCE`인 룸은 새로운 예약을 받을 수 없다.
- `status`를 `CLOSED` 또는 `MAINTENANCE`로 저장한 상태에서, 같은 룸에 대해 `endTime > 현재 가상 현재 일시`를 만족하는 `RESERVED` 또는 `CHECKED_IN` 예약이 한 건이라도 존재하면 의미 오류다.
- **잘못된 파일 대응 방식**: 중복 `roomId`, 비품 코드 중복, 운영 시간 역전, 허용되지 않은 상태값, 닫힌 룸에 대한 남아 있는 활성 예약이 발견되면 전체 파일을 무효로 판단하고 프로그램은 오류 메시지를 출력한 뒤 종료한다.

### 8.3 `reservations.txt`

- `reservationId`는 파일 전체에서 유일해야 한다.
- `roomId`는 `rooms.txt`에 존재해야 한다.
- `userId`는 `users.txt`에 존재하는 `member` 계정이어야 한다.
- 예약은 날짜를 넘길 수 없다. 하나의 레코드는 하나의 날짜 안에서만 유효하다.
- `startTime`은 `endTime`보다 빨라야 한다.
- `extensionCount`가 `0`이면 예약 길이는 60분 이상 240분 이하이며 30분 단위여야 한다.
- `extensionCount`가 `1`이면 최종 예약 길이는 90분 이상 270분 이하이며 30분 단위여야 한다. 또한 `endTime - 30분`으로 환산한 초기 길이가 60분 이상 240분 이하이어야 한다.
- 따라서 1차 범위에서 초기 예약은 최대 4시간이고, 연장 후 최종 사용 길이는 최대 4시간 30분이다.
- 모든 예약은 해당 룸의 일일 운영 시간 안에 완전히 포함되어야 한다.
- 같은 룸에서 `RESERVED` 또는 `CHECKED_IN` 상태인 예약끼리는 시간대가 겹치면 안 된다.
- 같은 사용자에 대해 `RESERVED` 또는 `CHECKED_IN` 상태인 예약끼리는 시간대가 겹치면 안 된다.
- 하나의 member가 가지는 미래 예약(`RESERVED`이면서 시작 시각이 현재 시각보다 늦은 예약)은 최대 2개여야 한다.
- `CHECKED_IN` 또는 `COMPLETED` 상태의 `checkedInAt`은 반드시 실제 체크인 시각이어야 하며, 예약 시작 10분 전 이상 예약 시작 후 15분 이하 구간 안에 있어야 한다.
- `RESERVED`, `CANCELLED`, `NO_SHOW` 상태의 `checkedInAt`은 반드시 `-`여야 한다.
- `extensionCount`가 `1`인 예약의 상태는 `CHECKED_IN` 또는 `COMPLETED`만 허용한다.
- 현재 시각보다 과거에 있는 `RESERVED` 예약이나 종료 시각이 지난 `CHECKED_IN` 예약은 파일 자체만으로는 의미 오류로 보지 않는다. 이들은 자동 상태 갱신 대상이다.
- **잘못된 파일 대응 방식**: 중복 ID, 존재하지 않는 룸 또는 사용자 참조, 잘못된 상태별 필드 조합, 30분 단위 위반, 운영 시간 위반, 겹침 예약, 미래 예약 개수 초과가 발견되면 전체 파일을 무효로 판단하고 프로그램은 오류 메시지를 출력한 뒤 종료한다.

### 8.4 `system_time.txt`

- 파일에는 정확히 하나의 `NOW` 레코드만 있어야 한다.
- 이 값이 시스템 전체의 유일한 현재 기준 시각이다.
- 실제 운영체제 시각은 읽지 않는다.
- 파일 값 자체는 과거일 수도 미래일 수도 있으나, 프로그램이 변경할 때는 기존 값보다 같거나 미래만 허용한다.
- **잘못된 파일 대응 방식**: 레코드 수가 1이 아니거나 형식이 잘못되면 파일 전체를 무효로 판단하고 프로그램은 오류 메시지를 출력한 뒤 종료한다.

## 9. 공통 입력 규칙

### 9.1 공통 처리 원칙

- 모든 입력은 한 줄 단위로 받는다.
- 입력 직후 선행 공백과 후행 공백을 한 번 제거한 뒤 검증한다.
- 공백 제거 후 빈 문자열이면 문법 오류로 처리한다.
- 현재 화면이 취소를 허용하는 경우에 한해 `0` 입력은 현재 기능 전체를 취소하고 이전 메뉴로 돌아간다.
- 한 필드가 잘못되면 그 필드만 다시 입력받는다.
- 문법 오류와 의미 오류 메시지는 구분한다.
  - 문법 오류 메시지 예시: `[입력 형식 오류] 날짜/시각은 yyyy-MM-dd HH:mm 형식으로 입력해야 합니다.`
  - 의미 오류 메시지 예시: `[입력 값 오류] 예약 시작 시각은 현재 시각 기준 14일 이내여야 합니다.`
- 재입력 횟수 제한은 두지 않는다.

### 9.2 사용자 ID

- **문법 규칙**: 4자 이상 12자 이하, 영문 소문자로 시작, 이후 영문 소문자와 숫자만 허용한다.
- **의미 규칙**
  - 회원가입 시에는 기존 사용자와 중복되면 안 된다.
  - 로그인, 패널티 초기화 대상 조회 시에는 존재하는 계정이어야 한다.
- **오류 메시지 예시**
  - 문법 오류: `[입력 형식 오류] 사용자 ID는 영문 소문자로 시작하는 4~12자여야 합니다.`
  - 의미 오류: `[입력 값 오류] 이미 사용 중인 사용자 ID입니다.`

### 9.3 비밀번호

- **문법 규칙**: 6자 이상 16자 이하, 영문 대/소문자, 숫자, `!@#$%^&*._-`만 허용하고 공백은 허용하지 않는다.
- **의미 규칙**
  - 로그인 시 저장된 비밀번호와 완전히 일치해야 한다.
  - 회원가입 시 비밀번호는 평문으로 저장되며 1차 범위에서 암호화하지 않는다.
- **오류 메시지 예시**
  - 문법 오류: `[입력 형식 오류] 비밀번호는 6~16자의 영문, 숫자, 지정 특수문자만 사용할 수 있습니다.`
  - 의미 오류: `[입력 값 오류] 비밀번호가 일치하지 않습니다.`

### 9.4 이름

- **문법 규칙**: 2자 이상 20자 이하, 한글·영문·공백 한 칸만 허용한다.
- **의미 규칙**
  - 앞뒤 공백은 허용하지 않는다.
  - 연속 공백 두 칸 이상은 허용하지 않는다.
  - 공백을 제외한 실제 문자가 최소 2자 이상이어야 한다.
- **오류 메시지 예시**
  - 문법 오류: `[입력 형식 오류] 이름은 2~20자의 한글 또는 영문으로 입력해야 합니다.`
  - 의미 오류: `[입력 값 오류] 이름에는 연속 공백을 사용할 수 없습니다.`

### 9.5 날짜/시각

- **문법 규칙**: 기본 입력 형식은 `yyyy-MM-dd HH:mm`이다.
- **의미 규칙**
  - 달력에 존재하는 날짜여야 한다.
  - 시와 분이 실제로 존재해야 한다.
  - 예약 시작/종료 입력은 분이 `00` 또는 `30`이어야 한다.
  - 예약 생성과 검색에서 시작 시각은 현재 시각보다 늦어야 한다.
  - 예약 생성에서 시작 시각은 현재 시각 기준 정확히 14일 후 같은 시각 이내여야 한다.
  - 예약 종료 시각은 시작 시각보다 늦어야 하며 같은 날짜여야 한다.
  - 현재 시각 변경 입력은 분 단위 전체를 허용하지만 기존 현재 시각보다 빠를 수 없다.
- **오류 메시지 예시**
  - 문법 오류: `[입력 형식 오류] 날짜/시각은 yyyy-MM-dd HH:mm 형식으로 입력해야 합니다.`
  - 의미 오류: `[입력 값 오류] 예약 시각은 30분 단위여야 합니다.`

### 9.6 룸 ID

- **문법 규칙**: `R` + 숫자 3자리 형식만 허용한다.
- **의미 규칙**
  - 존재하는 룸이어야 한다.
  - 예약 생성 시에는 `OPEN` 상태여야 한다.
- **오류 메시지 예시**
  - 문법 오류: `[입력 형식 오류] 룸 ID는 R101 형식으로 입력해야 합니다.`
  - 의미 오류: `[입력 값 오류] 존재하지 않는 룸 ID입니다.`

### 9.7 메뉴 선택

- **문법 규칙**: 메뉴에 표시된 번호와 정확히 같은 문자열만 허용한다.
- **의미 규칙**
  - 현재 화면에 없는 번호는 허용하지 않는다.
  - `01`처럼 표시된 번호와 다른 표기는 허용하지 않는다.
- **오류 메시지 예시**
  - 문법 오류: `[입력 형식 오류] 메뉴 번호만 입력해야 합니다.`
  - 의미 오류: `[입력 값 오류] 현재 메뉴에서 선택할 수 없는 번호입니다.`

### 9.8 잘못된 입력의 재입력 흐름

- 문법 오류가 발생하면 같은 입력 프롬프트를 다시 보여준다.
- 의미 오류가 발생하면 오류 이유를 함께 출력하고 같은 입력 프롬프트를 다시 보여준다.
- 다단계 입력 도중 현재 단계에서 `0`을 입력하면 전체 기능을 취소하고 이전 메뉴로 돌아간다.
- 기능이 성공하면 결과 메시지를 출력한 뒤 해당 역할의 메인 메뉴로 돌아간다.

## 10. 가상 현재 일시 처리 정책

### 10.1 기준 시각 정책

- 실제 시스템 시계는 사용하지 않는다.
- 현재 시각의 단일 기준은 `system_time.txt`의 `NOW` 레코드다.
- 이 값은 모든 사용자에게 동일하게 적용된다.
- member와 admin 모두 로그인 후 메인 메뉴에서 현재 시각 조회/변경 기능을 사용할 수 있다.

### 10.2 현재 시각 조회/변경 정책

- 기능 진입 시 현재 저장된 값을 먼저 보여준다.
- 변경 입력 화면 예시는 다음과 같다.
  - `현재 시각: 2026-03-10 13:00`
  - `새 현재 시각 입력(yyyy-MM-dd HH:mm, 취소는 0):`
- 새 현재 시각은 기존 현재 시각과 같거나 미래여야 한다.
- 과거 시각 입력은 의미 오류로 처리한다.
- 같은 시각을 입력하면 성공으로 처리하되 저장 파일은 다시 쓰지 않는다.
- 시각 변경 성공 시 변경 전 값과 변경 후 값을 함께 출력한다.
  - 예시: `[성공] 현재 시각이 2026-03-10 13:00에서 2026-03-10 18:00로 변경되었습니다.`

### 10.3 자동 상태 갱신 시점

자동 상태 갱신은 다음 시점에 반드시 수행한다.

1. 프로그램 시작 직후 전체 파일 검증이 끝난 직후
2. 로그인 성공 직후 역할별 메인 메뉴 진입 직전
3. 현재 시각 변경 직후
4. 예약 조회, 예약 생성, 예약 취소, 체크인, 사용 연장, 패널티 조회, 관리자 전체 예약 조회 실행 직전

CLI 프로그램은 실제 시간에 따라 백그라운드로 상태를 바꾸지 않는다. 상태 변화는 위 네 시점에서만 수행한다.

### 10.4 자동 상태 갱신 규칙

자동 상태 갱신은 예약 시작 일시 오름차순, 같은 시작 일시면 `reservationId` 오름차순으로 처리한다.

- `RESERVED` 상태이고 현재 시각이 `예약 시작 시각 + 15분`보다 늦으면 `NO_SHOW`로 변경한다.
- `NO_SHOW`로 변경될 때 사용자 패널티를 1점 증가시킨다.
- `CHECKED_IN` 상태이고 현재 시각이 예약 종료 시각과 같거나 늦으면 `COMPLETED`로 변경한다.
- `CANCELLED`, `COMPLETED`, `NO_SHOW`는 자동 상태 갱신으로 다시 다른 상태가 되지 않는다.
- 한 번 `NO_SHOW`로 바뀐 예약은 다시 체크인할 수 없다.
- 현재 시각이 크게 앞으로 이동해 여러 예약이 한 번에 `NO_SHOW`가 되면 각 예약마다 패널티 1점을 각각 부여한다.

### 10.5 현재 시각 변경이 예약에 미치는 영향

- 예약 시작 후 15분이 지난 `RESERVED` 예약은 자동으로 `NO_SHOW` 처리된다.
- 체크인된 예약은 종료 시각이 지나면 자동으로 `COMPLETED` 처리된다.
- 체크인 가능 구간 안에 있는 `RESERVED` 예약은 그대로 유지된다.
- 이미 `CANCELLED`, `NO_SHOW`, `COMPLETED`인 예약은 현재 시각 변경으로 다시 활성화되지 않는다.

## 11. 메뉴 구조

### 11.1 비로그인 메뉴

```text
[비로그인 메인 메뉴]
1. 회원가입
2. 로그인
0. 종료
```

- `1` 선택 시 회원가입 화면으로 이동한다.
- `2` 선택 시 로그인 화면으로 이동한다.
- `0` 선택 시 프로그램을 종료한다.

### 11.2 member 메인 메뉴

```text
[member 메인 메뉴]
1. 현재 시각 조회/변경
2. 스터디룸 목록 조회
3. 조건 기반 사용 가능 룸 검색
4. 예약 생성
5. 내 예약 조회
6. 예약 취소
7. 체크인
8. 사용 연장
9. 패널티 조회
0. 로그아웃
```

### 11.3 admin 메인 메뉴

```text
[admin 메인 메뉴]
1. 현재 시각 조회/변경
2. 스터디룸 목록 조회
3. 관리자용 전체 예약 조회
4. 관리자용 사용자 패널티 조회/초기화
5. 관리자용 룸 상태 변경
0. 로그아웃
```

### 11.4 하위 메뉴와 이동 흐름

- `현재 시각 조회/변경` 화면은 아래 하위 메뉴를 사용한다.

```text
[현재 시각 메뉴]
1. 현재 시각 조회
2. 현재 시각 변경
0. 이전 메뉴
```

- 회원가입, 로그인, 예약 생성, 예약 취소, 체크인, 사용 연장, 룸 상태 변경, 패널티 초기화 화면에서는 첫 입력 단계부터 `0`으로 취소할 수 있다.
- 조회 화면은 결과 출력 후 해당 역할의 메인 메뉴로 복귀한다.
- 하위 메뉴의 `0`은 이전 메뉴 복귀다.
- 역할 메인 메뉴의 `0`은 로그아웃이다.
- 비로그인 메인 메뉴의 `0`은 프로그램 종료다.

## 12. 사용 흐름

### 12.1 회원가입 후 예약 생성 흐름

1. 비로그인 메인 메뉴에서 회원가입 선택
2. `userId`, `password`, `name` 입력
3. `member` 계정 생성 후 비로그인 메인 메뉴 복귀
4. 로그인 수행
5. 현재 시각 확인
6. 사용 가능 룸 검색 또는 스터디룸 목록 조회
7. 예약 생성에서 룸 ID, 사용 인원, 시작 시각, 종료 시각 입력
8. 예약 성공 메시지 확인 후 member 메인 메뉴 복귀

### 12.2 체크인 후 사용 연장 흐름

1. member 로그인
2. 내 예약 조회로 오늘 예약 확인
3. 체크인 기능에서 예약 ID 입력
4. 현재 시각이 체크인 가능 구간이면 `CHECKED_IN` 처리
5. 사용 연장 기능에서 같은 예약 ID 입력
6. 뒤 예약이 없고 운영 종료 시각을 넘지 않으면 종료 시각을 30분 연장
7. 이후 현재 시각이 종료 시각 이상이 되면 자동으로 `COMPLETED` 처리

### 12.3 지연 취소 흐름

1. member 로그인
2. 예약 취소 메뉴 진입
3. 본인 `RESERVED` 예약 ID 입력
4. 현재 시각이 예약 시작 30분 전을 지났으면 `CANCELLED`로 바꾸고 패널티 1점 부여
5. 결과 메시지 출력 후 member 메인 메뉴 복귀

### 12.4 현재 시각 변경에 따른 자동 노쇼 흐름

1. 로그인 후 현재 시각 변경 메뉴 진입
2. 기존 시각과 새 시각을 함께 확인
3. 새 시각을 미래로 변경
4. 변경 직후 자동 상태 갱신 실행
5. 체크인 마감이 지난 `RESERVED` 예약은 `NO_SHOW`, 종료 시각이 지난 `CHECKED_IN` 예약은 `COMPLETED`로 변경
6. 필요한 경우 사용자 패널티가 즉시 누적됨

### 12.5 관리자 운영 흐름

1. admin 로그인
2. 전체 예약 조회로 현재 예약 상태 확인
3. 필요 시 특정 member의 패널티 조회 후 초기화
4. 룸 상태 변경 메뉴에서 예약이 남아 있지 않은 룸만 `CLOSED` 또는 `MAINTENANCE`로 변경
5. 조치 후 admin 메인 메뉴 복귀

## 13. 기능 명세

### 13.1 회원가입

- **기능 목적**: 새 `member` 계정을 생성한다.
- **입력값**: 사용자 ID, 비밀번호, 이름
- **사전조건**: 비로그인 상태여야 한다.
- **정상 동작 흐름**
  1. 사용자 ID 입력
  2. 비밀번호 입력
  3. 이름 입력
  4. 입력 문법 검증 및 중복 ID 검사
  5. `users.txt`에 `member`, 패널티 0, 상태 `ACTIVE`로 저장
  6. 성공 메시지 출력 후 비로그인 메인 메뉴 복귀
- **실패/예외 경우**
  - 사용자 ID 문법 오류
  - 비밀번호 문법 오류
  - 이름 문법 오류
  - 이미 존재하는 사용자 ID
  - 저장 직전 파일 재검증 실패
- **성공 시 변경되는 데이터**: `users.txt`에 사용자 1건 추가
- **실패 시 변경되지 않는 데이터**: `users.txt`, `rooms.txt`, `reservations.txt`, `system_time.txt`
- **사용자에게 보여줄 핵심 메시지 예시**: `[성공] 회원가입이 완료되었습니다. 로그인 후 이용해 주세요.`

### 13.2 로그인

- **기능 목적**: 사용자 인증 후 역할별 메인 메뉴로 진입한다.
- **입력값**: 사용자 ID, 비밀번호
- **사전조건**: 비로그인 상태여야 한다.
- **정상 동작 흐름**
  1. 사용자 ID 입력
  2. 비밀번호 입력
  3. 계정 존재 여부, 상태, 비밀번호 일치 여부 확인
  4. 로그인 성공 직후 자동 상태 갱신 수행
  5. 역할에 따라 member 또는 admin 메인 메뉴로 이동
- **실패/예외 경우**
  - 존재하지 않는 사용자 ID
  - 비밀번호 불일치
  - 허용되지 않은 사용자 상태
  - 로그인 직후 자동 상태 갱신 실패
- **성공 시 변경되는 데이터**: 기본적으로 영속 데이터 변경 없음. 단, 로그인 직후 자동 상태 갱신 결과에 따라 `reservations.txt`, `users.txt`가 갱신될 수 있다.
- **실패 시 변경되지 않는 데이터**: `users.txt`, `rooms.txt`, `reservations.txt`, `system_time.txt`
- **사용자에게 보여줄 핵심 메시지 예시**: `[성공] member 계정으로 로그인했습니다.`

### 13.3 로그아웃

- **기능 목적**: 현재 로그인 세션을 종료하고 비로그인 메인 메뉴로 돌아간다.
- **입력값**: 없음
- **사전조건**: 로그인 상태여야 한다.
- **정상 동작 흐름**
  1. 현재 세션 정보 제거
  2. 비로그인 메인 메뉴 출력
- **실패/예외 경우**: 없음
- **성공 시 변경되는 데이터**: 영속 데이터 변경 없음
- **실패 시 변경되지 않는 데이터**: `users.txt`, `rooms.txt`, `reservations.txt`, `system_time.txt`
- **사용자에게 보여줄 핵심 메시지 예시**: `[안내] 로그아웃되었습니다.`

### 13.4 현재 시각 조회/변경

- **기능 목적**: 공용 가상 현재 일시를 조회하거나 변경한다.
- **입력값**: 하위 메뉴 번호, 새 현재 시각(`yyyy-MM-dd HH:mm`)
- **사전조건**: 로그인 상태여야 한다.
- **정상 동작 흐름**
  1. 현재 시각 메뉴 출력
  2. 조회 선택 시 현재 값을 그대로 출력
  3. 변경 선택 시 기존 현재 시각을 먼저 보여줌
  4. 새 현재 시각 입력 후 문법/의미 검증
  5. 기존 값보다 같거나 미래면 `system_time.txt` 저장
  6. 저장 직후 자동 상태 갱신 수행
  7. 변경 전/후 시각과 갱신 결과를 출력하고 메인 메뉴 복귀
- **실패/예외 경우**
  - 메뉴 번호 오류
  - 날짜/시각 문법 오류
  - 기존 현재 시각보다 과거 입력
  - 저장 직전 파일 재검증 실패
  - 자동 상태 갱신 저장 실패
- **성공 시 변경되는 데이터**: `system_time.txt` 변경, 필요 시 `reservations.txt`와 `users.txt` 동시 갱신
- **실패 시 변경되지 않는 데이터**: `users.txt`, `rooms.txt`, `reservations.txt`, `system_time.txt`
- **사용자에게 보여줄 핵심 메시지 예시**: `[성공] 현재 시각이 2026-03-10 13:00에서 2026-03-10 18:00로 변경되었습니다.`

### 13.5 스터디룸 목록 조회

- **기능 목적**: 전체 룸의 기본 정보를 확인한다.
- **입력값**: 없음
- **사전조건**: 로그인 상태여야 한다.
- **정상 동작 흐름**
  1. `rooms.txt`의 모든 룸을 `roomId` 오름차순으로 조회
  2. 룸 ID, 수용 인원, 비품, 운영 상태, 운영 시간을 출력
  3. 메인 메뉴 복귀
- **실패/예외 경우**
  - 시작 시 로딩 이후 상태를 바꾸는 기능이 아니므로 추가 예외 없음
- **성공 시 변경되는 데이터**: 영속 데이터 변경 없음
- **실패 시 변경되지 않는 데이터**: `users.txt`, `rooms.txt`, `reservations.txt`, `system_time.txt`
- **사용자에게 보여줄 핵심 메시지 예시**: `[조회 결과] 총 5개의 스터디룸이 있습니다.`

### 13.6 조건 기반 사용 가능 룸 검색

- **기능 목적**: 특정 시간대에 예약 가능한 룸만 조건으로 걸러서 보여준다.
- **입력값**: 사용 인원, 시작 시각, 종료 시각, 필요 비품 목록(`-` 또는 `+` 구분 코드)
- **사전조건**: member 로그인 상태여야 한다.
- **정상 동작 흐름**
  1. 자동 상태 갱신 수행
  2. 사용 인원, 시작 시각, 종료 시각, 필요 비품 입력
  3. 예약 길이, 30분 단위, 14일 이내 여부 검증
  4. `OPEN` 상태이며 운영 시간 안에 들어가는 룸만 추림
  5. 수용 인원이 사용 인원 이상인 룸만 추림
  6. 필요 비품을 모두 포함한 룸만 추림
  7. 같은 시간대 `RESERVED` 또는 `CHECKED_IN` 예약이 없는 룸만 출력
- **실패/예외 경우**
  - member가 아닌 사용자의 접근
  - 날짜/시각 형식 오류
  - 사용 인원 범위 오류
  - 14일 범위 초과
  - 검색 결과 없음
  - 자동 상태 갱신 실패
- **성공 시 변경되는 데이터**: 영속 데이터 변경 없음
- **실패 시 변경되지 않는 데이터**: `users.txt`, `rooms.txt`, `reservations.txt`, `system_time.txt`
- **사용자에게 보여줄 핵심 메시지 예시**: `[조회 결과] 입력 조건을 만족하는 룸은 2개입니다.`

### 13.7 예약 생성

- **기능 목적**: member가 미래 시각의 스터디룸 예약을 생성한다.
- **입력값**: 룸 ID, 사용 인원, 시작 시각, 종료 시각
- **사전조건**: member 로그인 상태이고, 패널티가 2점 미만이어야 한다.
- **정상 동작 흐름**
  1. 자동 상태 갱신 수행
  2. 룸 ID, 사용 인원, 시작 시각, 종료 시각 입력
  3. 룸 존재 여부와 `OPEN` 상태 확인
  4. 사용 인원이 룸 수용 인원을 넘지 않는지 확인
  5. 시작/종료 시각이 30분 단위인지, 길이가 1시간 이상 4시간 이하인지 확인
  6. 시작 시각이 현재 시각보다 늦고 14일 이내인지 확인
  7. 룸 운영 시간 안에 포함되는지 확인
  8. 동일 룸 겹침 예약과 본인 겹침 예약 여부 확인
  9. 본인의 미래 예약 수가 2개 미만인지 확인
  10. 새 예약 ID를 생성하고 `RESERVED`, `checkedInAt=-`, `extensionCount=0`으로 저장
- **실패/예외 경우**
  - 패널티 2점 이상
  - 미래 예약 2개 보유
  - 존재하지 않는 룸 또는 닫힌 룸
  - 수용 인원 초과
  - 시작 시각이 현재보다 빠르거나 같음
  - 시작 시각이 14일 범위 초과
  - 예약 길이 제한 위반
  - 룸 겹침 또는 사용자 겹침 발생
  - 저장 직전 파일 재검증 실패
- **성공 시 변경되는 데이터**: `reservations.txt`에 예약 1건 추가
- **실패 시 변경되지 않는 데이터**: `users.txt`, `rooms.txt`, `reservations.txt`, `system_time.txt`
- **사용자에게 보여줄 핵심 메시지 예시**: `[성공] 예약이 생성되었습니다. 예약 ID는 rv0007입니다.`

### 13.8 내 예약 조회

- **기능 목적**: 로그인한 member의 예약 이력을 확인한다.
- **입력값**: 없음
- **사전조건**: member 로그인 상태여야 한다.
- **정상 동작 흐름**
  1. 자동 상태 갱신 수행
  2. 본인 `userId`에 해당하는 예약만 조회
  3. 날짜, 시작/종료 시각, 룸 ID, 상태, 체크인 시각, 연장 여부를 시작 시각 오름차순으로 출력
- **실패/예외 경우**
  - member가 아닌 사용자의 접근
  - 자동 상태 갱신 실패
- **성공 시 변경되는 데이터**: 영속 데이터 변경 없음. 단, 자동 상태 갱신 결과에 따라 `reservations.txt`, `users.txt`가 선반영될 수 있다.
- **실패 시 변경되지 않는 데이터**: `users.txt`, `rooms.txt`, `reservations.txt`, `system_time.txt`
- **사용자에게 보여줄 핵심 메시지 예시**: `[조회 결과] 내 예약은 총 3건입니다.`

### 13.9 예약 취소

- **기능 목적**: `RESERVED` 상태의 본인 예약을 취소한다.
- **입력값**: 예약 ID
- **사전조건**: member 로그인 상태이며, 대상 예약이 본인 소유의 `RESERVED` 상태여야 한다.
- **정상 동작 흐름**
  1. 자동 상태 갱신 수행
  2. 예약 ID 입력
  3. 본인 소유 여부와 현재 상태가 `RESERVED`인지 확인
  4. 현재 시각이 시작 시각 30분 전 이하이면 자유 취소 처리
  5. 현재 시각이 시작 시각 30분 전을 지난 경우 지연 취소 처리 및 패널티 1점 부여
  6. 예약 상태를 `CANCELLED`로 저장
- **실패/예외 경우**
  - 존재하지 않는 예약 ID
  - 본인 예약이 아님
  - 상태가 `RESERVED`가 아님
  - 자동 상태 갱신 후 이미 `NO_SHOW`로 변경됨
  - 저장 직전 파일 재검증 실패
- **성공 시 변경되는 데이터**: `reservations.txt` 상태 변경, 필요 시 `users.txt` 패널티 증가
- **실패 시 변경되지 않는 데이터**: `users.txt`, `rooms.txt`, `reservations.txt`, `system_time.txt`
- **사용자에게 보여줄 핵심 메시지 예시**: `[성공] 예약이 취소되었습니다. 지연 취소로 패널티 1점이 부여되었습니다.`

### 13.10 체크인

- **기능 목적**: 예약 사용 시작을 확정한다.
- **입력값**: 예약 ID
- **사전조건**: member 로그인 상태이며, 대상 예약이 본인 소유의 `RESERVED` 상태여야 한다.
- **정상 동작 흐름**
  1. 자동 상태 갱신 수행
  2. 예약 ID 입력
  3. 본인 예약인지, 상태가 `RESERVED`인지 확인
  4. 현재 시각이 예약 시작 10분 전 이상, 시작 후 15분 이하인지 확인
  5. 대상 룸이 `OPEN` 상태인지 확인
  6. 예약 상태를 `CHECKED_IN`으로 바꾸고 `checkedInAt`에 현재 시각 저장
- **실패/예외 경우**
  - 존재하지 않는 예약 ID
  - 본인 예약이 아님
  - 체크인 가능 구간보다 이르거나 늦음
  - 자동 상태 갱신 후 이미 `NO_SHOW`가 됨
  - 룸 상태가 `OPEN`이 아님
  - 저장 직전 파일 재검증 실패
- **성공 시 변경되는 데이터**: `reservations.txt`의 상태와 체크인 시각 변경
- **실패 시 변경되지 않는 데이터**: `users.txt`, `rooms.txt`, `reservations.txt`, `system_time.txt`
- **사용자에게 보여줄 핵심 메시지 예시**: `[성공] 체크인이 완료되었습니다.`

### 13.11 사용 연장

- **기능 목적**: 체크인된 예약의 종료 시각을 30분 한 번만 연장한다.
- **입력값**: 예약 ID
- **사전조건**: member 로그인 상태이며, 대상 예약이 본인 소유의 `CHECKED_IN` 상태이고 `extensionCount=0`이어야 한다.
- **정상 동작 흐름**
  1. 자동 상태 갱신 수행
  2. 예약 ID 입력
  3. 본인 예약인지, `CHECKED_IN` 상태인지, 아직 연장하지 않았는지 확인
  4. 새 종료 시각을 기존 종료 시각 + 30분으로 계산
  5. 새 종료 시각이 룸 운영 종료 시각을 넘지 않는지 확인
  6. 같은 룸에 새 종료 시각과 겹치는 뒤 예약이 없는지 확인
  7. 종료 시각을 갱신하고 `extensionCount`를 1로 저장
- **실패/예외 경우**
  - 존재하지 않는 예약 ID
  - 본인 예약이 아님
  - 상태가 `CHECKED_IN`이 아님
  - 이미 1회 연장함
  - 뒤 예약 존재
  - 룸 운영 종료 시각 초과
  - 룸 상태가 `OPEN`이 아님
  - 저장 직전 파일 재검증 실패
- **성공 시 변경되는 데이터**: `reservations.txt`의 종료 시각과 `extensionCount` 변경
- **실패 시 변경되지 않는 데이터**: `users.txt`, `rooms.txt`, `reservations.txt`, `system_time.txt`
- **사용자에게 보여줄 핵심 메시지 예시**: `[성공] 예약 종료 시각이 30분 연장되었습니다.`

### 13.12 패널티 조회

- **기능 목적**: 로그인한 member의 현재 패널티와 예약 생성 가능 여부를 확인한다.
- **입력값**: 없음
- **사전조건**: member 로그인 상태여야 한다.
- **정상 동작 흐름**
  1. 자동 상태 갱신 수행
  2. 본인 패널티 점수 출력
  3. 2점 미만이면 예약 생성 가능, 2점 이상이면 예약 생성 불가 메시지 출력
- **실패/예외 경우**
  - member가 아닌 사용자의 접근
  - 자동 상태 갱신 실패
- **성공 시 변경되는 데이터**: 영속 데이터 변경 없음. 단, 자동 상태 갱신 결과에 따라 `reservations.txt`, `users.txt`가 선반영될 수 있다.
- **실패 시 변경되지 않는 데이터**: `users.txt`, `rooms.txt`, `reservations.txt`, `system_time.txt`
- **사용자에게 보여줄 핵심 메시지 예시**: `[조회 결과] 현재 패널티는 2점이며 새로운 예약을 생성할 수 없습니다.`

### 13.13 관리자용 전체 예약 조회

- **기능 목적**: 모든 member 예약을 관리자가 한 번에 확인한다.
- **입력값**: 없음
- **사전조건**: admin 로그인 상태여야 한다.
- **정상 동작 흐름**
  1. 자동 상태 갱신 수행
  2. 모든 예약을 날짜, 시작 시각, 예약 ID 오름차순으로 조회
  3. 예약 ID, 사용자 ID, 룸 ID, 시작/종료 시각, 상태, 체크인 시각, 연장 여부를 출력
- **실패/예외 경우**
  - admin이 아닌 사용자의 접근
  - 자동 상태 갱신 실패
- **성공 시 변경되는 데이터**: 영속 데이터 변경 없음. 단, 자동 상태 갱신 결과에 따라 `reservations.txt`, `users.txt`가 선반영될 수 있다.
- **실패 시 변경되지 않는 데이터**: `users.txt`, `rooms.txt`, `reservations.txt`, `system_time.txt`
- **사용자에게 보여줄 핵심 메시지 예시**: `[조회 결과] 전체 예약 12건을 표시합니다.`

### 13.14 관리자용 사용자 패널티 초기화

- **기능 목적**: 특정 member의 패널티를 0점으로 초기화한다.
- **입력값**: 대상 사용자 ID
- **사전조건**: admin 로그인 상태여야 하며 대상 사용자는 존재하는 member여야 한다.
- **정상 동작 흐름**
  1. 대상 사용자 ID 입력
  2. 대상 계정 존재 여부와 역할 확인
  3. 현재 패널티를 확인
  4. 현재 패널티가 1 이상이면 0으로 변경
  5. 결과 메시지 출력 후 admin 메인 메뉴 복귀
- **실패/예외 경우**
  - 존재하지 않는 사용자 ID
  - admin 계정 지정
  - 이미 패널티가 0점인 사용자 지정
  - 저장 직전 파일 재검증 실패
- **성공 시 변경되는 데이터**: `users.txt`의 대상 사용자 패널티를 0으로 변경
- **실패 시 변경되지 않는 데이터**: `users.txt`, `rooms.txt`, `reservations.txt`, `system_time.txt`
- **사용자에게 보여줄 핵심 메시지 예시**: `[성공] u1001의 패널티가 0점으로 초기화되었습니다.`

### 13.15 관리자용 룸 상태 변경

- **기능 목적**: 관리자가 룸의 운영 상태를 `OPEN`, `CLOSED`, `MAINTENANCE` 중 하나로 변경한다.
- **입력값**: 룸 ID, 목표 상태
- **사전조건**: admin 로그인 상태여야 한다.
- **정상 동작 흐름**
  1. 룸 ID 입력
  2. 목표 상태 입력
  3. 룸 존재 여부와 현재 상태 확인
  4. 목표 상태가 `CLOSED` 또는 `MAINTENANCE`라면, 같은 룸에 대해 `endTime > 현재 시각`인 `RESERVED` 또는 `CHECKED_IN` 예약이 없는지 확인
  5. 조건을 만족하면 상태를 변경하고 저장
- **실패/예외 경우**
  - 존재하지 않는 룸 ID
  - 허용되지 않은 상태 입력
  - 현재 상태와 목표 상태가 같음
  - 남아 있는 현재/미래 예약 때문에 닫을 수 없음
  - 저장 직전 파일 재검증 실패
- **성공 시 변경되는 데이터**: `rooms.txt`의 대상 룸 상태 변경
- **실패 시 변경되지 않는 데이터**: `users.txt`, `rooms.txt`, `reservations.txt`, `system_time.txt`
- **사용자에게 보여줄 핵심 메시지 예시**: `[성공] R101의 운영 상태가 OPEN에서 MAINTENANCE로 변경되었습니다.`

## 14. 예외 처리 정책

### 14.1 입력 예외

- 문법 오류와 의미 오류를 반드시 구분해 출력한다.
- 잘못된 입력은 같은 프롬프트에서 다시 입력받는다.
- 현재 단계에서 `0`을 지원하는 화면은 언제든 취소 후 이전 메뉴로 복귀한다.
- 메뉴 번호 오류는 현재 메뉴를 다시 출력한다.

### 14.2 권한 예외

- member 전용 기능에 admin이 접근하면 즉시 거부한다.
- admin 전용 기능에 member가 접근하면 즉시 거부한다.
- 권한 예외는 데이터를 변경하지 않고 해당 역할 메인 메뉴로 복귀한다.

### 14.3 예약 규칙 예외

- 겹치는 룸 예약은 생성할 수 없다.
- 같은 사용자 겹침 예약은 생성할 수 없다.
- 미래 예약 2개 보유 시 추가 예약 생성은 거부한다.
- 패널티 2점 이상이면 예약 생성은 거부한다.
- `RESERVED`가 아닌 예약은 취소할 수 없다.
- 체크인 가능 구간 밖에서는 체크인할 수 없다.
- `CHECKED_IN`이 아니거나 이미 연장한 예약은 연장할 수 없다.

### 14.4 시간 예외

- 예약 시작/종료는 30분 단위가 아니면 거부한다.
- 예약 길이가 1시간 미만 또는 4시간 초과이면 거부한다.
- 예약 시작 시각이 현재 시각보다 늦지 않으면 거부한다.
- 예약 시작 시각이 현재 시각 기준 14일 범위를 넘으면 거부한다.
- 현재 시각 변경에서 과거 시각 입력은 거부한다.

### 14.5 파일 예외

- 접두어 오류, 필드 수 오류, 잘못된 이스케이프, 날짜 형식 오류는 문법 오류다.
- 중복 ID, 잘못된 참조, 예약 겹침, 미래 예약 수 초과, 닫힌 룸의 활성 예약 존재는 의미 오류다.
- 파일 오류는 부분 무시하지 않고 전체 로딩 실패로 처리한다.
- 시작 시 파일 오류가 있으면 프로그램은 종료한다.
- 저장 직전 파일 재검증 실패가 발생하면 현재 작업을 취소하고 프로그램을 종료한다.
- 저장 중 임시 파일 검증 실패 또는 교체 실패가 발생하면 원본 파일은 유지하고 프로그램을 종료한다.

### 14.6 메시지 정책

- 오류 메시지는 최소한 `오류 종류`, `대상`, `원인`을 포함한다.
- 파일 오류 메시지는 파일명과 행 번호를 포함한다.
  - 예시: `[파일 오류] reservations.txt 7행: 존재하지 않는 사용자 ID u9999를 참조합니다.`
- 성공 메시지는 핵심 결과값을 포함한다.
  - 예시: `[성공] 예약이 생성되었습니다. 예약 ID는 rv0012입니다.`

## 15. 상태 정의 및 상태 전이

### 15.1 예약 상태 정의

| 상태 | 의미 | 진입 조건 | 이탈 조건 |
|---|---|---|---|
| `RESERVED` | 예약만 생성된 상태 | 예약 생성 성공 | 취소, 체크인, 자동 노쇼 |
| `CANCELLED` | 사용자가 취소한 상태 | 예약 취소 성공 | 없음 |
| `CHECKED_IN` | 사용 시작이 확정된 상태 | 체크인 성공 | 자동 완료 |
| `COMPLETED` | 사용이 끝난 상태 | `CHECKED_IN` 상태에서 종료 시각 도달 | 없음 |
| `NO_SHOW` | 체크인 마감까지 미체크인 상태 | `RESERVED` 상태에서 시작 후 15분 경과 | 없음 |

### 15.2 예약 상태 전이 규칙

- `RESERVED -> CANCELLED`: 사용자가 취소하면 발생한다.
- `RESERVED -> CHECKED_IN`: 체크인 가능 구간 안에서 체크인하면 발생한다.
- `RESERVED -> NO_SHOW`: 자동 상태 갱신 시 현재 시각이 시작 후 15분보다 늦으면 발생한다.
- `CHECKED_IN -> COMPLETED`: 자동 상태 갱신 시 현재 시각이 종료 시각 이상이면 발생한다.
- `CHECKED_IN -> CHECKED_IN`: 연장 성공 시 상태는 유지되고 종료 시각만 변경된다.
- `CANCELLED`, `COMPLETED`, `NO_SHOW`는 종료 상태이며 다시 다른 상태로 바뀌지 않는다.

### 15.3 사용자 상태 정의

- 저장 상태는 1차 범위에서 `ACTIVE`만 사용한다.
- 예약 생성 가능 여부는 별도 상태값이 아니라 **패널티 기반 파생 조건**으로 판단한다.
  - 패널티 0~1점: 예약 생성 가능
  - 패널티 2점 이상: 예약 생성 불가
- 패널티가 2점 이상이어도 기존 예약 조회, 취소, 체크인, 연장은 규칙을 만족하면 수행할 수 있다.

### 15.4 룸 상태 정의

| 상태 | 의미 | 예약 생성 가능 여부 | 체크인 가능 여부 |
|---|---|---|---|
| `OPEN` | 정상 운영 | 가능 | 가능 |
| `CLOSED` | 예약 불가 상태 | 불가 | 불가 |
| `MAINTENANCE` | 점검 중 상태 | 불가 | 불가 |

### 15.5 룸 상태 전이 규칙

- `OPEN -> CLOSED`, `OPEN -> MAINTENANCE`: 해당 룸에 `endTime > 현재 시각`인 `RESERVED` 또는 `CHECKED_IN` 예약이 없을 때만 가능하다.
- `CLOSED -> OPEN`, `MAINTENANCE -> OPEN`: 언제든 가능하다.
- `CLOSED <-> MAINTENANCE`: 현재/미래 활성 예약이 없을 때만 가능하다.

- 예약 생성
- 내 예약 조회
- 예약 취소
- 체크인
- 30분 1회 연장
- 패널티 조회
- 관리자 전체 예약 조회
- 관리자 사용자 패널티 초기화
- 관리자 룸 상태 변경
- 일반 텍스트 파일 4종 읽기/쓰기/검증
- 자동 상태 갱신과 파일 직접 수정 오류 검출

1차 구현 범위에서는 admin 예약 생성, 복수 파일 포맷 버전 관리, 날짜별 점검 시간표, 대기열, 통계 화면을 넣지 않는다.

## 17. 2차 확장 범위

2차 확장 범위는 1차 기능과 명확히 분리한다.

- **대기열 기능**: 원하는 시간대의 룸이 없을 때 대기 신청을 받고, 예약 취소 시 우선순위에 따라 전환
- **통계 기능**: 룸별 이용률, 사용자별 예약/노쇼 통계, 패널티 발생 통계
- **점검 시간 등록 기능**: 룸 상태를 단순 현재값이 아니라 날짜·시간 구간 단위로 관리
- **세부 검색 강화**: 시간 범위 유사 검색, 장비 조합 추천, 최소 공실 간격 검색, 정렬 옵션 추가
- **운영 감사 이력**: 현재 시각 변경 이력, 관리자 조치 이력 저장

## 18. 검사 관점 핵심 항목

기획 단계에서 반드시 확인할 핵심 검사 항목은 다음과 같다.

1. `users.txt`에 `USER` 외 접두어가 들어가면 문법 오류로 검출되는가.
2. `rooms.txt`에 필드 수가 7개가 아니면 문법 오류로 검출되는가.
3. `reservations.txt`에 잘못된 이스케이프가 있으면 문법 오류로 검출되는가.
4. `system_time.txt`에 두 줄 이상 존재하면 오류로 종료하는가.
5. 빈 줄이 포함된 파일을 오류로 처리하는가.
6. `users.txt`에 admin 계정이 하나도 없으면 의미 오류로 검출되는가.
7. admin 계정의 패널티가 0이 아니면 의미 오류로 검출되는가.
8. `rooms.txt`에서 `openTime >= closeTime`이면 의미 오류로 검출되는가.
9. `rooms.txt`에서 비품 코드가 중복되면 의미 오류로 검출되는가.
10. `reservations.txt`에서 존재하지 않는 `roomId`를 참조하면 의미 오류로 검출되는가.
11. `reservations.txt`에서 admin `userId`를 참조하면 의미 오류로 검출되는가.
12. 같은 룸의 `RESERVED` 예약 두 건이 겹치면 의미 오류로 검출되는가.
13. 같은 member의 `RESERVED` 예약 두 건이 겹치면 의미 오류로 검출되는가.
14. 한 member의 미래 예약이 3건이면 의미 오류로 검출되는가.
15. 예약 시작/종료 시각이 30분 단위가 아니면 거부되는가.
16. 예약 길이가 1시간 미만이면 거부되는가.
17. 예약 길이가 4시간을 초과하면 거부되는가.
18. 연장 후 총 길이가 4시간 30분을 넘을 수 없도록 막는가.
19. 현재 시각보다 이르거나 같은 시작 시각으로 예약 생성이 거부되는가.
20. 현재 시각 기준 14일을 초과한 시작 시각이 거부되는가.
21. 체크인이 시작 10분 전보다 빠르면 거부되는가.
22. 체크인이 시작 후 15분을 초과하면 자동 노쇼 뒤 거부되는가.
23. 예약 시작 정확히 30분 전 취소는 패널티 없이 처리되는가.
24. 예약 시작 29분 전 취소는 패널티 1점으로 처리되는가.
25. 자동 노쇼 시 패널티가 정확히 1점만 증가하는가.
26. 자동 상태 갱신이 같은 예약에 대해 중복 패널티를 부여하지 않는가.
27. 패널티가 2점이 되면 새로운 예약 생성만 막고 기존 예약 조회/취소/체크인은 허용하는가.
28. `CHECKED_IN`이 아닌 예약은 연장할 수 없도록 막는가.
29. 뒤 예약이 있으면 연장이 거부되는가.
30. 룸 운영 종료 시각을 넘는 연장이 거부되는가.
31. `CLOSED` 또는 `MAINTENANCE` 룸에서는 체크인이 거부되는가.
32. 현재/미래 예약이 남아 있는 룸은 `CLOSED` 또는 `MAINTENANCE`로 변경되지 않는가.
33. 현재 시각 변경 시 변경 전 값과 변경 후 값이 함께 출력되는가.
34. 과거 시각으로 현재 시각 변경을 시도하면 의미 오류로 처리되는가.
35. 현재 시각을 크게 앞으로 이동했을 때 `NO_SHOW`와 `COMPLETED`가 규칙대로 일괄 갱신되는가.
36. 수동 편집으로 파일 참조가 깨진 경우 프로그램이 부분 로딩하지 않고 종료하는가.
37. 저장 직전 파일 재검증에 실패하면 기존 원본 파일이 유지되는가.
38. 메뉴 선택에서 `01` 같은 입력을 허용하지 않는가.
39. 이름 입력의 선행/후행 공백이 제거되고 빈 값이 되면 오류로 처리되는가.
40. 로그인 실패 시 어떠한 파일도 변경되지 않는가.

## 19. 기능 단위 분업 예시

5인 팀 예시는 아래와 같이 기능 묶음 단위로 나눈다. 각 인원은 자기 기능 묶음에 대해 **기획 세부화, 구현, 검사 시나리오 작성, 발표 준비**를 모두 일부씩 담당한다.

1. **팀원 A - 계정 및 공통 입력**
   - 회원가입, 로그인, 로그아웃
   - `users.txt` 검증
   - 공통 입력 검증 및 오류 메시지 정리
   - 시연 시 계정 관련 흐름 발표 준비
2. **팀원 B - 가상 현재 일시 및 자동 상태 갱신**
   - 현재 시각 조회/변경
   - `system_time.txt` 검증
   - 자동 `NO_SHOW`/`COMPLETED` 처리
   - 시간 경계 테스트와 시연 장면 준비
3. **팀원 C - 룸 데이터 및 검색**
   - 스터디룸 목록 조회
   - 조건 기반 사용 가능 룸 검색
   - `rooms.txt` 검증
   - 룸 상태와 운영 시간 관련 시연 준비
4. **팀원 D - 예약 수명주기 1**
   - 예약 생성
   - 내 예약 조회
   - 예약 취소
   - `reservations.txt` 생성 규칙과 충돌 검사 테스트 준비
5. **팀원 E - 예약 수명주기 2 및 관리자 기능**
   - 체크인
   - 사용 연장
   - 패널티 조회
   - 관리자 전체 예약 조회, 사용자 패널티 초기화, 룸 상태 변경
   - 관리자 시연 시나리오와 예외 사례 발표 준비

팀원 수가 줄면 1번과 2번, 4번과 5번을 합친다. 팀원 수가 늘면 파일 검증, 시간 정책, 관리자 기능을 추가로 분리한다.

## 20. 구현 범위 제한 및 제외 사항

- 언어는 Java로 고정한다.
- 인터페이스는 표준 입력/출력 기반 CLI로 고정한다.
- GUI, 화살표키 기반 TUI는 구현하지 않는다.
- 실제 시스템 시간, 외부 API, 크롤링은 사용하지 않는다.
- 저장은 일반 텍스트 파일만 사용하고 JSON, CSV, DBMS, 직렬화는 사용하지 않는다.
- 멀티스레딩, IPC, 다중 프로세스 동기화는 구현하지 않는다.
- 관리자 예약 생성 기능은 1차 범위에서 제외한다.
- 대기열, 통계, 점검 시간 등록, 세부 검색 강화는 2차 범위에서만 다룬다.
- 비밀번호 암호화 저장, 권한 세분화, 감사 로그 저장은 1차 범위에서 제외한다.
