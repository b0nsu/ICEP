## 1. 검사 개요

본 보고서는 1차 기획서 원판을 기준으로 한 전체 통합 검사 결과만을 정리한다. 검사 강의자료 #6의 지침에 따라, 단위 검사나 중간 단계 통합 검사는 보고 대상에서 제외하고 최종 완제품 CLI 프로그램에 대한 테스트 케이스 기반 검사만 수행하였다.

검사 목표는 다음 두 가지다.

1. 기획서에 명시된 사용 흐름과 예외 처리 분기를 빠짐없이 확인한다.
2. 각 입력 범위의 정상값, 경계값, 비정상값을 포함해 데이터 커버리지를 확보한다.

## 2. 검사 환경 및 방법

### 2.1 검사 환경

- 운영체제: Windows 11
- Java: JDK 17
- 빌드 도구: Gradle Wrapper
- 데이터 위치: 프로젝트 루트의 `data/`

### 2.2 검사 방법

검사 수행은 전체 통합 검사 방식으로 진행하였다.

- `cmd /c gradlew.bat clean check --no-daemon`로 컴파일, 회귀 검사, `check` 작업까지 일괄 수행
- 사용자 흐름 확인은 `gradlew run`을 기준으로 수행하고, 파일 상태 변화는 `data/*.txt`로 확인
- 결과 확인은 화면 출력과 `data/*.txt` 변경 여부를 함께 사용

검사 강의자료 #6의 지침에 따라, 너무 당연한 입력 방법과 결과 확인 방법은 표에서 생략하거나 간략화하였다. 실제 결과가 예상 결과와 동일한 경우에는 `(좌동)`으로 표기하였다.

## 4. 종합 결론

보고 대상 전체 통합 테스트 케이스 24개를 수행했고, 재검사 기준으로 모두 예상 결과와 실제 결과가 일치하였다.

- 전체 통합 검사만 보고 대상에 포함
- 기획서 기반 주요 사용 흐름과 입력 예외 처리 분기를 모두 포함
- 정상값, 경계값, 비정상값을 섞어 데이터 커버리지를 확보

따라서 본 구현물은 1차 기획서 원판에 명시된 범위 내에서 높은 확률로 올바르게 동작한다고 판단한다. 다만 검사 강의자료의 원칙대로, 이 결론은 어디까지나 수행한 테스트 케이스 범위에 대한 확률적 판단이며, 미검사 입력 조합 전부에 대한 절대적 보증은 아니다.
